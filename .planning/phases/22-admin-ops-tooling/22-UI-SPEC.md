## Typography

Type scale uses Tailwind defaults; only sizes/weights actually rendered in this phase are declared. System font stack inherited.

| Role | Size | Weight | Line Height | Tailwind class | Used for |
|------|------|--------|-------------|----------------|----------|
| Body | 14px | 400 (regular) | 1.5 (`leading-normal`) | `text-sm` | Card body text, metric values, sidebar labels, error messages, recent-error timestamps |
| Label | 12px | 400 (regular) | 1.4 (`leading-tight`) | `text-xs` | Badge text (`Badge` component default), `text-muted-foreground` metric labels (`waiting`, `active`, etc.), "Updated Ns ago" chip, version tag |
| Heading | 18px | 500 (medium) | 1.4 | `text-lg font-medium` | Section headings: "Queues", "Workers", "Recent Errors", "Modules" (matches `system/health.tsx:148`) |
| Display | 24px | 500 (medium) | 1.3 | `text-2xl font-medium` | Page title (`<h1>` "Job Monitor", "System Health") (matches `system/health.tsx:81,99,120`) |

**Two weights only:** `400 regular` for body and labels; `500 medium` for headings and display. The medium-on-display + medium-on-headings pairing keeps the admin chrome calm and matches the Phase 7/11 accessibility audit baseline already shipped — display gets its prominence from size (24px) and `space-y-6` rhythm rather than a heavier weight.

> **Note on 12 vs 14px:** the `text-xs` (12px) label and `text-sm` (14px) body sizes are only 2px apart, which on its own is a weak differentiator. Differentiation is augmented by **role + color**: labels always carry `text-muted-foreground` (the `--muted-foreground` token), body text uses default `--foreground`. This pairing — smaller-and-muted vs default-and-foreground — is the established admin-app pattern and is sufficient to disambiguate the two roles for sighted users; AT users get the same disambiguation via badge text and surrounding markup.

## Color

The admin dashboard uses shadcn neutral baseColor (zero saturation greyscale + a single red destructive hue). HSL tokens listed below resolve via `--*` CSS variables in `packages/ui/src/styles/globals.css`.

| Role | Token | Light value | Usage |
|------|-------|-------------|-------|
| Dominant (60%) | `--background` | `hsl(0 0% 100%)` (white) | Page background, main content surface, card surfaces (`--card` resolves to same hue) |
| Secondary (30%) | `--sidebar` / `--muted` / `--secondary` | `hsl(0 0% 98%)` sidebar / `hsl(0 0% 96.1%)` muted | Sidebar background, card sub-surfaces, badge `secondary` variant background, skeleton pulse |
| Accent (10%) | `--primary` | `hsl(0 0% 9%)` (near-black) | Active sidebar item background, primary button bg, "healthy" badge bg via `Badge variant="default"`, focus ring |
| Destructive | `--destructive` | `hsl(0 84.2% 60.2%)` (red) | "Unhealthy" / "Critical" / "Dead" status badges, retry-failure error card, "Failed to load" copy emphasis |

**60/30/10 split per page:**
- 60% — page background + card body surfaces (white).
- 30% — sidebar (#fafafa), card grid backgrounds, secondary badges (`Degraded`, `Stale`, `Warning`), skeleton placeholders.
- 10% — `Healthy` status badges (Badge `default` variant — solid near-black), `Refresh` button hover, sidebar active-item highlight, focus rings.

**Accent reserved for** (explicit list — never "all interactive elements"):
1. Active sidebar item ring/background (`SidebarMenuButton isActive`).
2. Primary "Healthy" / "Connected" badges (`Badge` default variant).
3. Page focus rings (`--ring` already drives Phase 11 a11y focus-visible discipline).
4. Hover state of the "Refresh now" button on `/system` page (Button `variant="outline"`'s hover).

**Destructive reserved for** (explicit list):
1. "Unhealthy" overall-status badge.
2. "Critical" queue-depth badge (waiting > 1000).
3. "Dead" worker-heartbeat badge (age > 5× interval).
4. DB `unhealthy` badge (DB disconnected or lag probe failed).
5. Retry button on the page-level error card stays `variant="outline"` (NOT destructive) to match existing `system/health.tsx:87`.

**Status → Badge variant mapping** (single source of truth — extends `getStatusVariant` in `system/health.tsx:38-53`):

| Status string | Badge variant | Meaning |
|---------------|--------------|---------|
| `healthy` | `default` (solid primary) | All good |
| `degraded` | `secondary` | Soft warning, no action required immediately |
| `warning` | `secondary` | Queue waiting > 100 (CONTEXT D-09) |
| `stale` | `secondary` | Worker heartbeat 2-5× interval old |
| `unhealthy` | `destructive` | Page or contributor in failure state |
| `critical` | `destructive` | Queue waiting > 1000 |
| `dead` | `destructive` | Worker heartbeat > 5× interval or missing |
| `unknown` | `outline` | Module loaded but no contributor (D-16 default fallback for transient races) |

No new color tokens introduced. Do **not** add a new "warning yellow" or "info blue" — the neutral palette intentionally collapses these into `secondary` to keep the admin UI calm.

### Standard primary CTA

This phase has no destructive user actions. The only CTAs are **non-destructive read-only operations**:
- "Reload dashboard" (errors on `/jobs`)
- "Refresh now" (`/system` page — manual refetch trigger; auto-poll continues at 30s).
- "Retry" (page-level error fallback — uses existing `common.retry`).

No confirmation dialogs are introduced in this phase. (Bull-board's own retry/promote/remove actions live inside the iframe and are gated behind `BULL_BOARD_READ_ONLY=false`, which is an env-level operator decision, not a UI control. Out of scope for this UI-SPEC.)

### `/system` route — `/health/detailed` consumer (replaces existing)

Section ordering (top to bottom; matches priority for an on-call operator's eye):

1. Page header: `<h1 class="text-2xl font-medium">{t("systemHealth.title")}</h1>` + right-aligned chip group (`Updated Ns ago` + `RefreshCw` button).
2. **Overall status card** (existing pattern, kept) — large `Badge` + status sentence.
3. **Queues** grid — `Layers` icon + h2; per-queue card shows: name, badge, waiting/active/delayed/completed/failed (4-col tight grid), thresholds row.
4. **Workers** grid — `Cpu` icon + h2; per-worker card shows: instanceId, queues (badge list), `Last heartbeat: 12s ago`, status badge.
5. Two-column row:
   - **Database** — `Database` icon + h2; rows: connection (`connected`/`disconnected` badge), round-trip latency (`23 ms` value, `lagUnavailable` if null).
   - **Recent Errors** — `AlertTriangle` icon + h2; scrollable list (max-height `max-h-72 overflow-y-auto`), each row: timestamp · source-tag · truncated message · dedup count.
6. **Modules** grid — `Boxes` icon + h2; per-module card: name, loaded badge, status badge, `details` JSON in `<pre>` if present (collapsible via `<details><summary>`).

Polling chip implementation:
```tsx
<div class="flex items-center gap-2">

    {ago === 0 ? t("systemHealth.updatedJustNow") : t("systemHealth.updatedAgo", { time: agoLabel })}

```

Per-queue threshold visualization:
- Inline progress-style bar via Tailwind divs, **no chart library**: a `div` with `bg-muted` background and a colored fill `div` inside. Width = `min(waiting / critical, 1) * 100%`. Color: `bg-foreground` (healthy), `bg-secondary-foreground` (warning), `bg-destructive` (critical).
- Thresholds row: `Warn at 100 · Critical at 1000` in `text-xs text-muted-foreground`.

Worker freshness display:
- `lastHeartbeat` rendered via `formatDistanceToNow(new Date(lastHeartbeat), { addSuffix: true })` from `date-fns` (already imported in `system/health.tsx`).
- Status badge to the right of the relative time, mapped via the Status table in the Color section.
