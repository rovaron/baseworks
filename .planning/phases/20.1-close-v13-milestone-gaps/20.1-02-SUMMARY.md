# Phase 20.1 Plan 02: Billing scopedDb API correction Summary

**The v1.3 production TypeError was not an import-resolution bug — it was 7 handlers calling `scopedDb.select()` with no argument. Empirically excluded the original D-07 hypothesis via the D-05 probe, then routed all 7 handlers through the canonical `scopedDb.select(table)` form per user-authorized Option A. New API-level regression test mounts real billingRoutes against real scopedDb so the handler↔wrapper interaction is now CI-locked.**

## Accomplishments

- **Hypothesis A excluded by probe.** D-05 probe (commit `13a69ff`) printed `billingCustomers loaded: object true` — the symbol IS defined at module-eval time. The original plan's D-07 import-path fix would have been a no-op.
- **Empirical root cause identified.** All 7 billing `ctx.db.*` handlers call `ctx.db.select()` with no argument and then `.from(billingCustomers).where(eq(billingCustomers.tenantId, ctx.tenantId))`. But `ctx.db` is the `scopedDb` wrapper, whose `select(table)` REQUIRES the table arg and auto-injects the tenant predicate. Calling `.select()` passes `undefined` to Drizzle's `from()`, which then crashes inside `getTableColumns(undefined)` → `undefined[Table.Symbol.Columns]` is the production TypeError observed in v1.3 UAT.
- **Option A applied uniformly.** All 7 ctx.db handlers now use `ctx.db.select(billingCustomers).limit(1)` — the canonical scopedDb API. The redundant manual `WHERE tenantId = ...` predicate is dropped because `scopedDb.select` auto-injects it.
- **D-05 probe stripped.** `console.log` block removed from `get-subscription-status.ts` together with the now-unused `eq` import.
- **API-level regression gate.** `apps/api/__tests__/billing-subscription.test.ts` (116 lines) mounts the real `billingRoutes` plugin against a probe Elysia app with `handlerCtx.db = scopedDb(realDb, testTenantId)`. Asserts the no-billing-row case returns HTTP 200 with `{ success: true, data: { status: "inactive", hasSubscription: false, providerSubscriptionId: null, providerPriceId: null, currentPeriodEnd: null } }`. Verified RED before the fix (500 from the TypeError), GREEN after.
- **Mock-context preserved for 159 existing tests.** Updated `select` mock to a thenable that supports BOTH the new `.select(t).limit()` shape AND the legacy chain. Zero per-test edits required; 89 billing + 70 auth/example unit tests still GREEN.

## Deviations from Plan

### Rule 4 / Option A architectural decision

**1. [Rule 4 - Architectural] Original plan premise was empirically wrong**
- **Found during:** Task 1 (D-05 probe execution before this continuation run)
- **Issue:** The plan's D-07 hypothesis (workspace re-export indirection causes `billingCustomers` to be `undefined` at handler invocation) was disproved by the probe. The bug is a different shape entirely — scopedDb API misuse.
- **Fix:** User authorized Option A from the Rule 4 checkpoint: route all 7 ctx.db handlers through `scopedDb.select(table)` and drop the redundant manual tenant predicate.
- **Files:** plan revision in `20.1-02-PLAN.md` `<revision_log>` section (commit `639674f`); 7 handlers + mock-context updated (commit `c57adfb`).

### Auto-fixed Issues

**2. [Rule 1 - Bug] sync-usage.ts mis-listed in continuation prompt**
- **Found during:** Pre-Task-3 reconnaissance
- **Issue:** The continuation prompt listed `packages/modules/billing/src/jobs/sync-usage.ts` as the 8th affected handler, but `sync-usage.ts` uses a raw `createDb(env.DATABASE_URL)` instance (not scopedDb). Its `db.select({...}).from(...)` is the correct raw-Drizzle API and not buggy. Patching it would have BROKEN it (raw drizzle select doesn't accept a table arg the way scopedDb does).
- **Fix:** Excluded from the refactor. Documented in `20.1-02-PLAN.md` `<revision_log>` and the GREEN commit body (`c57adfb`).
- **Files:** N/A (non-action — explicit exclusion).

**3. [Rule 1 - Bug] Original test skeleton's `INSERT INTO organization` was unnecessary and FK-incorrect**
- **Found during:** Task 2 test authoring
- **Issue:** The plan's Task 2 skeleton seeded an `organization` row before hitting the route. But `billing_customers.tenantId` is a plain `varchar(36)` with no FK constraint to `organization.id` (verified via `packages/db/src/schema/base.ts:tenantIdColumn`). Seeding the org row was needless ceremony AND would have left dangling rows on test failure since `afterAll` only cleans up if `canConnect`.
- **Fix:** Test uses a fresh UUID per run as the test tenantId without any organization-table seeding. `afterAll` cleans up by deleting any `billing_customers` row with that tenantId (defensive — the test never inserts one, but the cleanup is cheap insurance).
- **Files:** `apps/api/__tests__/billing-subscription.test.ts` (commit `8e74107`).

**4. [Rule 1 - Bug] Plan skeleton's import path `@baseworks/module-billing/routes` is not a valid subpath export**
- **Found during:** Task 2 test authoring
- **Issue:** `packages/modules/billing/package.json` only declares `main: "./src/index.ts"`; no subpath exports. `import { billingRoutes } from "@baseworks/module-billing/routes"` would have failed module resolution.
- **Fix:** Use the package default export and read `.routes` off the `ModuleDefinition`: `import billingModule from "@baseworks/module-billing"; const billingRoutes = billingModule.routes;`.
- **Files:** `apps/api/__tests__/billing-subscription.test.ts` (commit `8e74107`).
