## Implementation Decisions

### Drizzle migration baseline (Gap 1)

- **D-01:** **Reset migration history → single fresh `0000_*.sql` baseline.** Delete `packages/db/migrations/0001_rename_stripe_to_provider.sql`, `packages/db/migrations/meta/_journal.json`, and `packages/db/migrations/meta/0000_snapshot.json`. Run `bun run db:generate` against the current `@baseworks/db` schema (with `provider_*` columns already defining the live shape) → produces a single `0000_<slug>.sql` baseline + matching meta entries that drizzle-kit's own tooling generates. The historical `stripe_*` → `provider_*` rename SQL has nowhere to apply on a fresh DB and is intentionally lost — the codebase is a starter-kit fork-and-customize template; preserving an incomplete history adds no value and trips fork users.
- **D-02:** **Existing local DBs reset via `docker compose down -v`.** Document the reset step in onboarding (`README.md` and/or `apps/api/README.md` — planner picks the canonical home). New baseline assumes a fresh empty DB; existing devs nuke their volume and rerun `bun run db:migrate`. Aligns with the starter-kit mental model and avoids machinery for a fork-template project.
- **D-03:** **CI-grade migration regression test.** Add `bun test` that boots a fresh PG (PGlite or testcontainers — planner picks) and asserts `db:migrate` exits 0 and creates the expected tables. Locks SC#1 as a regression gate, not just an operator observation. Path TBD by planner; likely `packages/db/src/__tests__/fresh-clone-migrate.test.ts` adjacent to existing drizzle integration tests.
- **D-04:** **Delete `0001_rename_stripe_to_provider.sql` outright.** No archive, no idempotent guard. The new 0000 baseline reflects the post-rename column shape; the rename SQL is redundant.

### obsContext / OTel traceId bridge (Gap 3)

- **D-10:** **Bridge direction: obsContext → OTel context (not the reverse).** Critical clarification from codebase scout: there is **no `@opentelemetry/instrumentation-http` registered** in this codebase (deps in lockfile, but no `registerInstrumentations({...})` call exists in `packages/observability/src/`). The "server span" today is created manually in `apps/api/src/core/middleware/observability.ts:87` via `getTracer().startSpan(...)` — defaulting to Noop. Therefore the original todo's "seed obsContext from OTel server span" framing inverts what's possible: there is no auto-instrumented OTel span to read from. The fix flows obsContext into OTel context so any downstream `tracer.startSpan` (server span in observabilityMiddleware, producer span in `wrap-queue`) inherits obsContext's trace_id, and `propagation.inject(context.active(), carrier)` writes that trace_id into BullMQ carriers.
- **D-11:** **Mechanism: synthetic OTel SpanContext at Bun.serve fetch boundary.** In `apps/api/src/index.ts` Bun.serve fetch wrapper, BEFORE `obsContext.run(...)`, build a synthetic `SpanContext` from the `traceId`/`spanId` values produced (or adopted from inbound `traceparent`) by the trust-gate logic. Wrap the existing handler call:
  ```ts
  const otelCtx = trace.setSpanContext(ROOT_CONTEXT, { traceId, spanId, traceFlags: TraceFlags.SAMPLED, isRemote: false });
  return context.with(otelCtx, () =>
    obsContext.run({ requestId, traceId, spanId, locale, ... }, () => app.handle(req))
  );
  ```
  All downstream spans created via the Tracer port inherit this trace_id; `propagation.inject` writes it into queue carriers; worker `wrapProcessorWithAls` extracts it and seeds its own `obsContext.traceId` to match — closing the loop end-to-end.
- **D-12:** **Drop `OBS_TRUST_TRACEPARENT_FROM` and the CIDR-based trust gate.** OTel HTTP propagation's default posture (always honor inbound `traceparent`) is acceptable for v1.3. Phase 19 deferred UAT item 1 ("real-gateway TCP-peer CIDR-trusted traceparent adoption") is implicitly resolved by removal. `decideInboundTrace` becomes a thin helper that parses inbound `traceparent` if present (adopt traceId+spanId); otherwise mints fresh ids. The `OBS_TRUST_TRACEPARENT_HEADER` and `OBS_TRUST_TRACEPARENT_FROM` env vars are removed from `packages/config/src/env.ts` and from `validateObservabilityEnv()`. Update `19-VALIDATION.md` deferred item 1 status to "resolved-by-removal in Phase 20.1." Trust posture for production hardening is a future phase concern.
- **D-13:** **Phase 20 D-06 producer span preserved.** The `{queue.name} publish` PRODUCER span with full OTEL messaging semantic conventions stays exactly as Phase 20 D-06 specifies. The bridge fix only changes which trace_id the span inherits at creation time (now: obsContext.traceId via D-11's `context.with`).
- **D-14:** **Test gate extends Phase 20's `apps/api/__tests__/observability-bullmq-trace.test.ts`.** Add an assertion: producer-side log line `traceId === T_PRODUCER`, worker-side log line `traceId === T_PRODUCER`, AND extracted `traceparent` from carrier carries `T_PRODUCER`. Folds into the existing Phase 20 D-08 E2E test rather than a separate file. Locks SC#3 as merge gate.

<specifics>
## Specific Ideas

- **Bridge direction is the inverse of the todo's framing.** The todo (`2026-04-26-bridge-obscontext-traceid-to-otel-server-span.md`) writes "seed obsContext from OTel server span" assuming `instrumentation-http` is creating one. Codebase scout confirms it is not. Phase 20.1 D-10/D-11 flow obsContext → OTel context instead. Downstream agents must read the todo for problem context but follow D-10/D-11 for the mechanism.
- **Synthetic OTel SpanContext is the bridge primitive.** Build `{ traceId, spanId, traceFlags: SAMPLED, isRemote: false }` from `decideInboundTrace`'s output and pass it to `trace.setSpanContext(ROOT_CONTEXT, ...)` — clean, no special integration with any tracer adapter. Works whether the Tracer port is Noop or wired to a real OTel SDK exporter (Phase 21 territory).
- **Trust posture relaxation is intentional.** Dropping `OBS_TRUST_TRACEPARENT_FROM` (D-12) accepts OTel's default "always honor inbound traceparent" for v1.3. Production-grade trust hardening returns as a future concern. Phase 19's deferred CIDR-trusted UAT item closes by removal, not by implementation.
- **Migration history is intentionally lost.** D-01 produces a new 0000 baseline matching the current `provider_*` schema; the historical `stripe_*` → `provider_*` rename SQL is deleted. Acceptable for a starter-kit fork-and-customize template; would be unacceptable for a production line-of-business app with deployed environments, but Baseworks isn't one.
