### WR-02: Billing handlers swallow non-`Error` throws via `error.message || ...`

**Files modified:**
- `packages/modules/billing/src/commands/cancel-subscription.ts`
- `packages/modules/billing/src/commands/change-subscription.ts`
- `packages/modules/billing/src/commands/create-checkout-session.ts`
- `packages/modules/billing/src/commands/create-one-time-payment.ts`
- `packages/modules/billing/src/commands/create-portal-session.ts`
- `packages/modules/billing/src/queries/get-billing-history.ts`

**Commit:** `1dc69dc`

**Applied fix:** Tightened all 6 catch blocks from `error: any` to `error: unknown` with `instanceof Error` narrowing:

```ts
} catch (error: unknown) {
  const message =
    error instanceof Error ? error.message : "Failed to <op>";
  return err(message || "Failed to <op>");
}
```

Behavior preserved for the common case (real Error with truthy `.message`). Non-Error throws (thrown strings, provider errors with undefined `.message`) now fall back to the per-handler default message instead of producing `undefined` or a property-access TypeError. This also resolves the Biome `noExplicitAny` concern (IN-03), so IN-03 is implicitly closed by this commit.

The wider helper-extraction refactor suggested in WR-02 (shared `toErrResult()` in `@baseworks/shared` plus rethrow-vs-swallow policy) was intentionally **not** applied — it crosses package boundaries, changes wrapCqrsBus interaction semantics, and is genuinely cross-cutting. Per the fix-run instruction "if this scope feels too broad for an auto-fix, mark as skipped" I scoped this to the safe narrowing fix and left the helper extraction for a deliberate future phase.

**Verification:** `bun test packages/modules/billing` → **89 pass / 0 fail / 291 expect() calls**.

### WR-04: ALS seed does not capture `set-cookie` ergonomics for the locale fallback

**Files modified:**
- `apps/api/src/lib/locale-cookie.ts`
- `apps/api/src/lib/__tests__/locale-cookie.test.ts`
- `apps/api/src/index.ts`

**Commit:** `702006f`

**Applied fix:**

1. New helper `hasNextLocaleCookie(cookieHeader: string | null): boolean` in `apps/api/src/lib/locale-cookie.ts`. Detects whether the raw Cookie header contains a `NEXT_LOCALE=...` segment regardless of value validity. Anchored regex `(?:^|;\s*)NEXT_LOCALE=` to avoid suffix-only collisions like `MY_NEXT_LOCALE=...`.
2. In `apps/api/src/index.ts` Bun.serve fetch wrapper:
   - Capture `parsedLocale` separately so both the `locale ?? defaultLocale` fallback and the `localeCookieNeedsClear` decision share one parse.
   - After `app.handle(req)` resolves, when `parsedLocale === null && hasNextLocaleCookie(cookieHeader)`, append `Set-Cookie: NEXT_LOCALE=; Max-Age=0; Path=/` to the response headers using `headers.append` (preserves any existing auth/session Set-Cookie). Falls back to reconstructing the Response if `response.headers` is immutable.
3. 7 new unit tests in `apps/api/src/lib/__tests__/locale-cookie.test.ts` covering null/empty inputs, malformed values (`%ZZ`), unsupported locales (`xyzzy`), embedded position among other cookies, and the suffix-collision guard.

This addresses BOTH the malformed-escape case (D-16 / H-01) and the unsupported-locale case — both leave the user stuck on defaultLocale forever without an explicit cookie clear.

**Verification:** `bun test apps/api/src/lib/__tests__/locale-cookie.test.ts` → **17 pass / 0 fail**. Full `bun test apps/api` → **146 pass / 1 fail**; the single failure is `Worker entrypoint > worker starts without HTTP server and logs startup`, which was confirmed pre-existing by re-running the suite against a stash that excluded the WR-04 changes (138 pass / 1 fail, same failure). Unrelated to this fix.
