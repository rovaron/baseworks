No critical issues. Net assessment: ship-ready for v1.3.

## Warnings

### WR-01: Always-trust inbound `traceparent` is a known correlation-injection surface

**File:** `apps/api/src/lib/inbound-trace.ts:9-24`
**Issue:** The Phase 20.1 D-12 rewrite drops the prior CIDR + trusted-header gate and now adopts any well-formed inbound `traceparent` from any client. The block comment correctly flags this ("revisit for production hardening"), but a public-internet-facing API that honors arbitrary inbound trace IDs lets a hostile client (a) join its requests to an existing internal trace by guessing/replaying a traceId, and (b) poison Tempo/Sentry trace correlation by emitting traffic with attacker-chosen 32-hex IDs. This is mitigated for v1.3 because the API is not yet public, but it is a real surface the moment ingress is opened.
**Fix:** Track this in a follow-up issue (e.g., "Production trust hardening for inbound traceparent" already noted in `20.1-CONTEXT.md`). Concrete v2 shape:
```ts
// Trust only when request originates from a known ingress hop OR carries a
// signed gateway header. Default to fresh ids otherwise.
export function decideInboundTrace(
  req: Request,
  opts: { trustedFromHeader?: string; trustedCidrs?: string[] },
): { traceId: string; spanId: string } {
  const inbound = req.headers.get("traceparent");
  const trusted =
    (opts.trustedFromHeader && req.headers.get(opts.trustedFromHeader)) ||
    isFromTrustedCidr(req, opts.trustedCidrs);
  if (inbound && trusted) {
    const m = TRACEPARENT_RE.exec(inbound);
    if (m) return { traceId: m[1], spanId: m[2] };
  }
  return mintFresh();
}
```
Until then, document the deferral in `ROADMAP.md` (or the security-hardening section of the v1.4 milestone) so it is not lost.

### WR-02: Billing handlers swallow non-`Error` throws via `error.message || ...`

**File:** `packages/modules/billing/src/commands/cancel-subscription.ts:46-48` (and 6 sibling handlers — `change-subscription.ts:43-45`, `create-checkout-session.ts:49-51`, `create-one-time-payment.ts:50-52`, `create-portal-session.ts:50-52`, `queries/get-billing-history.ts:44-46`)
**Issue:** Every handler uses the same pattern:
```ts
} catch (error: any) {
  return err(error.message || "Failed to ...");
}
```
Three concerns: (1) `error: any` defeats TS narrowing — a thrown non-Error (e.g., a Stripe `StripeError` whose `.message` is sometimes undefined, or a thrown string) silently falls through to the generic message and the original cause is dropped from the result; (2) provider-side error objects often carry `.code` / `.type` / `.statusCode` fields that callers may want to inspect (refund vs. invalid-card vs. rate-limit), all flattened to a single string here; (3) since Phase 19 added `wrapCqrsBus(... getErrorTracker())`, throwing would be observed by the error tracker — these `try/catch` blocks short-circuit that path and leave the failure silent in Sentry/GlitchTip.
**Fix:** Tighten the catch and either rethrow non-Stripe errors or attach a code:
```ts
} catch (error: unknown) {
  const message = error instanceof Error ? error.message : "Failed to cancel subscription";
  // Optional: preserve structured provider info
  // const code = (error as { code?: string }).code;
  return err(message);
}
```
For the broader pattern, consider a shared `toErrResult(error, fallback)` helper in `@baseworks/shared` so all 7 handlers use one tested codepath. This is the same kind of cross-cutting refactor that paid off in Phase 19 (`wrapCqrsBus`).

### WR-03: `mock-context.ts` `select` thenable resolves to the SAME `selectResult` regardless of `.limit(n)` / `.where(...)` chain

**File:** `packages/modules/__test-utils__/mock-context.ts:30-47`
**Issue:** The `buildSelectThenable` helper returns the same `selectResult` array for every chain shape — `await db.select(t)`, `await db.select(t).limit(1)`, `await db.select().from(t).where(...)`, etc. Tests that only care about "first row" semantics work fine, but a test that exercises a real handler issuing two different selects in sequence (different table args, different limits) will see identical data for both. Currently no Phase 20.1 test trips this because handlers issue exactly one `select(billingCustomers).limit(1)`, but the next module to add multi-select handlers will silently get phantom data.
**Fix:** Either (a) document the limitation in the helper docstring so future authors know to reach for a real `scopedDb` (per the Phase 20.1 SC#2 regression test pattern in `apps/api/__tests__/billing-subscription.test.ts`), or (b) make `results.select` accept a function so different chains return different shapes:
```ts
const selectResolver =
  typeof results?.select === "function" ? results.select : () => results?.select ?? [];
// then in .limit(n) / .where(...) etc., call selectResolver({ limit: n, where: ... }).
```
Low priority — the SC#2 test already showed that integration-scope tests with the real wrapper are the right tool for handler-vs-scopedDb interaction.

### WR-04: `apps/api/src/index.ts` ALS seed does not capture `set-cookie` ergonomics for the locale fallback

**File:** `apps/api/src/index.ts:184-186`
**Issue:** When the inbound `NEXT_LOCALE` cookie is malformed (D-16/H-01 hardening just landed), `parseNextLocaleCookie` returns null and the request proceeds with `defaultLocale`. The browser, however, still has the bad cookie set — every subsequent request will repeat the silent fallback and the user is stuck on `defaultLocale` forever with no signal. Not a Phase 20.1 regression (the original code had the same behavior), but the new try/catch in `locale-cookie.ts:26-32` is the natural place to tag this for follow-up.
**Fix:** Optional v1.3.1 hardening: when the decode fails, set a response `Set-Cookie: NEXT_LOCALE=; Max-Age=0; Path=/` to clear the bad cookie. This is best done in the Bun.serve fetch wrapper since `parseNextLocaleCookie` is purely synchronous and shouldn't acquire a Response side-effect:
```ts
// In Bun.serve fetch wrapper:
const cookieHeader = req.headers.get("cookie");
const parsed = parseNextLocaleCookie(cookieHeader);
const localeNeedsClear =
  cookieHeader?.includes("NEXT_LOCALE=") && parsed === null;
const locale = parsed ?? defaultLocale;
// ... after app.handle(req) returns Response, append clearing cookie if needed.
```
Document as known-issue in v1.3 release notes if not addressed before milestone close.

### IN-01: Drizzle config uses non-null assertion on `process.env.DATABASE_URL`

**File:** `packages/db/drizzle.config.ts:15`
**Issue:** `url: process.env.DATABASE_URL!` will produce a confusing error from drizzle-kit ("Cannot read properties of undefined") rather than a clear "DATABASE_URL is required" message when the env is missing. Drizzle-kit typically runs in `bun run db:*` scripts that don't go through `@baseworks/config` env validation.
**Fix:**
```ts
const databaseUrl = process.env.DATABASE_URL;
if (!databaseUrl) {
  throw new Error("DATABASE_URL is required for drizzle-kit. Set it in .env or via shell.");
}
export default defineConfig({
  // ...
  dbCredentials: { url: databaseUrl },
});
```

### IN-02: Unused destructuring of `_input` in `cancel-subscription` and `getSubscriptionStatus`

**Issue:** Both handlers receive `_input` (underscore prefix denotes intentionally unused) — this is correct convention, but the input schema is `Type.Object({})` which produces `{}` at runtime, so the parameter is never useful. Could be inlined or omitted if `defineCommand`/`defineQuery` support a no-input overload.

### IN-03: `error: any` cast bypasses TS strict mode in 6 billing handlers

**File:** All 5 billing commands + `get-billing-history.ts` query
**Issue:** Same as WR-02 but flagged separately for the linter angle: `error: any` will fail Biome's `noExplicitAny` rule if it is enabled at any point. With Biome listed in the project stack and described as a future enforcement target, this pattern needs to migrate to `error: unknown` with type guards.
**Fix:** See WR-02 for the full pattern. Could be addressed in a single sweep with the helper-extraction refactor.

### IN-04: `mock-context.ts:13-18` `results` parameter uses untyped `any[]`

**File:** `packages/modules/__test-utils__/mock-context.ts:13-18`
**Issue:** `results?: { select?: any[]; insert?: any[]; update?: any; delete?: any }` defeats the test-helper's type-safety contract. Tests that pass the wrong shape (e.g., a plain object instead of an array for `select`) won't get a TS error.
**Fix:** Type each field generically:
```ts
export function createMockDb<TRow = unknown>(results?: {
  select?: TRow[];
  insert?: TRow[];
  update?: Partial<TRow>;
  delete?: TRow | undefined;
}) { ... }
```
Low priority — test helpers traditionally tolerate looser typing.

### IN-05: `apps/api/src/index.ts:139` `(ctx: any)` derive cast loses Eden Treaty type fidelity

**File:** `apps/api/src/index.ts:139`, `apps/api/src/index.ts:162`
**Issue:** The `.derive` and `.delete("/api/tenant", ...)` callbacks use `(ctx: any)`. The CLAUDE.md explicitly calls out "No `as any` casts" as an Eden Treaty type-safety invariant for the chain. While `(ctx: any)` is technically not `as any`, it has the same effect — the derived `tenantId`/`userId`/`requestId` fields don't surface in the inferred `App` type, so consumers get `unknown`/`any` for these. Phase 20.1 didn't author this code (it predates the phase), but the surrounding edits in `Bun.serve fetch` make the file part of the review surface.
**Fix:** Define a typed context shape:
```ts
type TenantCtx = { tenantId: string; userId: string; requestId: string };
.derive({ as: "scoped" }, (ctx: TenantCtx) => { ... })
```
Or use Elysia's `t.Object(...)` schema attached upstream so types propagate. Follow-up issue, not a Phase 20.1 blocker.

### IN-06: `fresh-clone-migrate.test.ts:26` URL-rewrite regex doesn't preserve credentials with embedded slashes

**File:** `packages/db/src/__tests__/fresh-clone-migrate.test.ts:26`
**Issue:** `ADMIN_DB_URL.replace(/\/[^/?]+(\?|$)/, '/${SCRATCH_DB_NAME}$1')` matches the last path segment, which works for `postgres://user:pass@host:5432/baseworks` but would mishandle a URL with an additional path component (e.g., a connection-pooler URL like `postgres://user:pass@host:5432/baseworks/options`). Currently fine because the codebase only uses bare DB-name URLs, but worth a comment.
**Fix:** Add a docstring note that the helper assumes a bare-DB-name connection string, or use the `URL` constructor for robustness:
```ts
const u = new URL(ADMIN_DB_URL);
u.pathname = `/${SCRATCH_DB_NAME}`;
const scratchUrl = u.toString();
```
