## 2026-04-26 — Post-Probe Architectural Decision (Rule 4)

### Empirical findings from Task 1

The D-05 probe (committed `13a69ff`) printed `billingCustomers loaded: object true` —
`billingCustomers` IS defined at module-eval time. **Hypothesis A (D-07 / import-path
indirection) is EXCLUDED.**

### Actual root cause

The 7 `ctx.db.*` billing handlers call `ctx.db.select()` with NO argument and then chain
`.from(table).where(eq(table.tenantId, ctx.tenantId))`. But `ctx.db` is the `scopedDb`
wrapper (`packages/db/src/helpers/scoped-db.ts:46-50`), whose `select` signature is:

```ts
select(table: any) {
  return db.select().from(table).where(eq(table.tenantId, tenantId)) as any;
}
```

Calling `scopedDb.select()` with no arg passes `undefined` to the inner Drizzle call
`db.select().from(undefined)`, which then tries `getTableColumns(undefined)` →
`undefined[Table.Symbol.Columns]` → the production TypeError observed in v1.3 UAT.

`sync-usage.ts` (the 8th handler the parent prompt listed) is NOT affected: it uses a raw
`createDb(env.DATABASE_URL)` instance, not scopedDb. Its `db.select({...}).from(...)` is the
correct raw-Drizzle API.

### User decision (Option A)

User authorized Option A from the Rule 4 checkpoint: scope all 7 ctx.db handlers uniformly
through `scopedDb.select(table)`. The transformation per call site:

```ts
// before
await ctx.db.select().from(billingCustomers).where(eq(billingCustomers.tenantId, ctx.tenantId)).limit(1);

// after
await ctx.db.select(billingCustomers).limit(1);
```

The redundant tenant predicate is dropped because `scopedDb.select` injects it
automatically. Non-tenant predicates would need composition via `.raw`, but no such
predicates exist in any of the 7 affected handlers.

### Affected handlers (all 7 take Option A)

1. `packages/modules/billing/src/queries/get-subscription-status.ts:37`
2. `packages/modules/billing/src/queries/get-billing-history.ts:30`
3. `packages/modules/billing/src/commands/cancel-subscription.ts:29`

### Mock-context update

The shared `packages/modules/__test-utils__/mock-context.ts:24-31` mocks `select` as a
chained `select().from().where().limit()` builder. After Option A the existing unit tests
call `ctx.db.select(table).limit()`, so the mock's `select()` must return a thenable that
ALSO supports `.limit(...)` directly while keeping back-compat for any other call shape.
