## obsContext/OTel traceId bridge approach

### Q: Which bridge approach should we lock in?

| Option | Description | Selected |
|--------|-------------|----------|
| Option A — seed obsContext.traceId from OTel server span (Recommended) | Read trace.getActiveSpan() in fetch wrapper; minimal scope | ✓ |
| Option B — remove obsContext.traceId entirely; pino mixin reads OTel each call | Cleaner long-term; bigger blast radius across Phase 19 invariants | |
| Option C — document the duality, accept it | Cheapest; worst long-term; blocks SC#3 | |

**User's choice:** Option A — seed obsContext.traceId from OTel server span (Recommended)

**Notes:** The codebase scout that followed this answer revealed a critical correction: there is no `@opentelemetry/instrumentation-http` registered in this codebase. The "server span" today is created manually via `getTracer().startSpan(...)` in `observabilityMiddleware.derive`. So the actual bridge has to flow obsContext → OTel context (not the reverse). The user's "Option A spirit" was preserved (align obsContext with OTel context, drop independent ID minting); the mechanism was sharpened in the next question.
