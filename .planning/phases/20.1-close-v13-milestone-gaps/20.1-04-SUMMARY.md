## Decisions Made

- **Plan 04 layered cleanly on Plan 03's nested wrapper.** Plan 03 had already established `context.with(otelCtxWithReqSpan, () => obsContext.run(seed, fn))` at the Bun.serve fetch boundary, exactly as CONTEXT.md D-20 default merge order anticipated. H-02 (D-17) layered into the same Bun.serve fetch handler scope by replacing one statement; H-03 (D-18) layered into the inner `obsContext.run` callback by converting it from `() => app.handle(req)` to `async () => { try { ... } catch { ... } }`. No rebase needed.
- **`AsyncLocalStorage.run` accepts an async callback.** Confirmed via Node's `node:async_hooks` API contract: `run(store, callback, ...args)` invokes the callback synchronously and returns whatever the callback returns. When the callback is async, the returned promise carries the ALS frame as long as the await chain stays unbroken — which it does inside `app.handle(req)`. The single-ALS-seed invariant from Phase 19 D-12 is preserved structurally.
- **`@opentelemetry/api`'s `SpanStatusCode` was already a transitive dep.** No `package.json` change needed; the named-import line at apps/api/src/index.ts:30-37 already pulled `context`, `ROOT_CONTEXT`, `SpanContext`, `TraceFlags`, `trace` from the same module — `SpanStatusCode` slotted in alphabetically.
- **H-02 test design — duck-typed Request stand-in.** Bun's `Headers` constructor rejects newlines in header values, so `new Request("...", { headers: { "x-request-id": "foo\n[CRITICAL]" } })` throws `TypeError` before our helper can see the value. The helper is defined against the structural surface `{ headers: { get(name): string | null } }`, so the test injects a duck-typed object with a custom `headers.get` — exercising the validator's CRLF defense independently of upstream Headers normalization. Documented in a JSDoc above the helper.
- **ROADMAP.md line-number fix reversed from plan body's prediction.** The plan body assumed line 89 was the stale cell (`0/0`) and line 58 was current (`0/4`). Reality at HEAD was the inverse: line 89 had been auto-bumped to `3/4` by Plan 03's plan-close `gsd-sdk roadmap.update-plan-progress`, while line 58 was still stale at `0/4`. Fix: amended line 58 → `3/4`. The SDK call at this plan's close bumps both to `4/4`.

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Bun's strict `Headers` constructor rejects raw newlines, blocking the H-02 newline regression test as originally written**

- **Found during:** Task 2 GREEN — running the new test against the new helper.
- **Issue:** The plan body's test code uses `new Request("...", { headers: { "x-request-id": "foo\n[CRITICAL] fake" } })` to drive the newline-rejection path. Bun's strict `Headers` constructor throws `TypeError: Header 'x-request-id' has invalid value` before the constructor returns, so the helper never sees the newline value — the test fails on construction rather than on validator logic.
- **Fix:** Added a `reqWithRawHeader(name, raw)` helper that returns a duck-typed `Request` stand-in: `{ headers: { get(n) { return n.toLowerCase() === name.toLowerCase() ? raw : null; } } } as unknown as Request`. The newline test now uses this helper; all other tests continue to use the real `Request` constructor where the value is valid. The duck-typed surface matches the helper's structural type contract; no production code changes.
