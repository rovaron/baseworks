# Phase 20.1 Plan 03: obsContext ↔ OTel Bridge + CIDR Trust-Gate Removal Summary

**Synthetic OTel SpanContext built at the Bun.serve fetch boundary so producer-side and consumer-side log lines for the same BullMQ-enqueueing request carry an identical traceId; CIDR-based traceparent trust gate dropped (D-12) for OTel's always-trust default.**

## Accomplishments

- `apps/api/src/index.ts` Bun.serve fetch wrapper now wraps `obsContext.run(...)` inside `context.with(otelCtxWithReqSpan, ...)` where `otelCtxWithReqSpan` carries a synthetic OTel `SpanContext` built from the same `{traceId, spanId}` the ALS frame is seeded with. Downstream `tracer.startSpan` (HTTP server span at `apps/api/src/core/middleware/observability.ts:87`, producer span in `packages/observability/src/wrappers/wrap-queue.ts`) and `propagation.inject(context.active(), carrier)` (BullMQ carrier traceparent) all naturally inherit `obsContext.traceId` end-to-end. SC#3 closed at the production-code level; manual milestone UAT confirms the operator-visible behaviour ("API and worker log lines for the same request carry an identical traceId").
- `apps/api/src/lib/inbound-trace.ts` shrank from 115 lines to 24 lines. The CIDR-based trust gate (`TRUSTED_CIDRS`, `TRUSTED_HEADER`, `remoteAddr` parameter, three-dot canonicality guard) and the `inboundCarrier` return field are gone. The helper is now a thin parse-or-fresh wrapper: regex-parse a well-formed inbound `traceparent` and adopt it; otherwise mint fresh ids.
- `packages/config/src/env.ts` lost `OBS_TRUST_TRACEPARENT_FROM`, `OBS_TRUST_TRACEPARENT_HEADER`, and the entire ~50-line CIDR-validation block inside `validateObservabilityEnv()`. The `import ipaddr from "ipaddr.js"` line was removed alongside. The function-shape `switch` arms for `ERROR_TRACKER`, `TRACER`, and `METRICS_PROVIDER` are preserved verbatim.
- `apps/api/__tests__/observability-bullmq-trace.test.ts` extended with the D-14 lock-in regression gate: producer-log `traceId === T_PRODUCER`, consumer-log `traceId === T_PRODUCER`, and `recordedData._otel.traceparent` matches `^00-${T_PRODUCER}-[0-9a-f]{16}-[0-9a-f]{2}$`. Lock-in regression — NOT a TDD RED, the in-process simulation already wraps with `context.with` independent of production code.
