#### 1. SC#3 Production traceId continuity

Compare the `traceId` field in both lines.

**Expected:** Both the API log line and the worker log line carry an identical `traceId` value (the same 32-hex string). The BullMQ job carrier `_otel.traceparent` should also contain the same 32-hex traceId prefix.

**Why human:** The `context.with(otelCtxWithReqSpan, () => obsContext.run(...))` bridge is wired in `apps/api/src/index.ts` and `propagation.inject(context.active(), ...)` is present in `wrap-queue.ts`. The in-process simulation harness in `observability-bullmq-trace.test.ts` (D-14 lock-in assertions) confirms the tracing chain is correct within the simulation. However, verifying that `context.active()` at the `wrap-queue.ts` call site is actually the synthetic SpanContext set by the Bun.serve fetch wrapper — and not a disconnected root context due to an async boundary — requires running the actual request-to-worker flow with live processes.
