<objective>
Bridge `obsContext.traceId` to the OTel context at the Bun.serve fetch boundary so producer-side (API) and consumer-side (BullMQ worker) log lines for the same request carry an identical `traceId`. Drop the CIDR-based traceparent trust gate (D-12) — OTel's default "always honor inbound traceparent" is acceptable for v1.3. Closes Phase 20.1 SC#3.

Purpose: Phase 20's Human UAT test 2 surfaced a duality. `obsContext.traceId` (used by the pino mixin for log lines) and the OTel ambient context's trace_id (used by `propagation.inject` writing to BullMQ carriers, and by `tracer.startSpan` creating producer spans) were minted independently per request. The producer span and the carrier `traceparent` ended up with one trace_id; the API/worker log lines carried a different one. CONTEXT.md D-10 inverts the original todo's framing: there is no auto-instrumented HTTP server span to seed obsContext from, so the bridge flows obsContext to OTel context (D-11). A synthetic OTel SpanContext built at the Bun.serve fetch boundary makes the OTel ambient context inherit obsContext's traceId; downstream `tracer.startSpan` and `propagation.inject` then naturally use the same trace_id end-to-end.

Output:
- `apps/api/src/index.ts` Bun.serve fetch wrapper wraps `obsContext.run(...)` inside `context.with(otelCtxWithReqSpan, ...)` (D-11)

    export function decideInboundTrace(req: Request): { traceId: string; spanId: string } {
      const inbound = req.headers.get("traceparent");
      if (inbound) {
        const m = TRACEPARENT_RE.exec(inbound);
        if (m) return { traceId: m[1], spanId: m[2] };
      }
      return {
        traceId: crypto.randomUUID().replace(/-/g, "").slice(0, 32),
        spanId: crypto.randomUUID().replace(/-/g, "").slice(0, 16),
