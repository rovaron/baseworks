## What was built

Phase 25 turned the Phase 24 throwing scaffolds into three working `FileStorage`
adapters proven equivalent by one shared conformance suite, plus the supporting
test infrastructure (HMAC local signing, CORS validator, deterministic fixture
set) that the signed-upload phases (26–28) build on.

1. **Shared conformance suite** — `runFileStorageConformance(label, makeStorage)`
   in `packages/storage/src/adapters/__tests__/conformance.ts` (named `.ts`, not
   `.test.ts`, so `bun test` never runs it standalone). Nine backend-agnostic
   behaviors covering round-trip byte-equality, authoritative `stat`
   (byteSize + mimeType), idempotent delete, signed-URL minting/expiry, and the
   Pitfall-1 assertion that no `storageKey`/`key` field appears in the typed
   result. Consumed by all three adapter test files (D-25-04).

2. **Local (filesystem) adapter** — HMAC-SHA256 signed URLs minted by
   `adapters/local/signing.ts` (`signLocalUrl` + `verifyLocalSignature`,
   timing-safe compare via `node:crypto.timingSafeEqual`). FS object I/O with a
   `.meta.json` sidecar so `stat` returns an authoritative `mimeType`, plus a
   path-traversal guard on every method. The placeholder endpoint path
   `/api/files/local/:bucket/:key` is minted/verified here but served in Phase 26
   (D-25-01).

3. **S3 + S3-compat adapters** — both delegate to one shared
   `adapters/s3/shared-s3.ts` built on `Bun.S3Client` (no `@aws-sdk/*` dep). The
   two classes differ only in client construction: S3 uses `region`; S3-compat
   uses `endpoint` + `virtualHostedStyle: false` to honor `S3_FORCE_PATH_STYLE`
   (D-25-03, R3). `signUpload` returns `method: "PUT"` for all three adapters;
   POST-policy is deferred (D-25-02).

4. **CORS validator + templates** — `scripts/validate-cors.ts`
   (`validateCorsConfig`) asserts no wildcard origins, `ETag` in `ExposeHeaders`,
   and `PUT` present; wired as root script `bun run validate-cors`. Four committed
   templates under `docs/integrations/file-storage/cors/`
   (`aws-s3`, `r2`, `minio`, `garage`), each using concrete origins.

5. **Deterministic fixture set** — five artifacts under
   `packages/storage/__test-fixtures__/` (baseline 100×100, photo 5000×5000, bomb
   50000×50000, truncated, SVG-with-script) + `manifest.json`, generated by
   `scripts/generate-fixtures.ts` (hand-built PNGs via `node:zlib.deflateSync`,
   no `Math.random`). `test-support/fixtures.ts` (`loadFixture`) is the
   consumption seam: conformance uses the baseline PNG today; Phase 28 transform
   tests consume all five (D-25-06).

## Divergences from the contract (accurate record)

- **CI harness location:** the contract proposed a separate
  `.github/workflows/storage-conformance.yml`. Implementation instead folded the
  MinIO service container into the existing `ci` job of `validate.yml` (same
  effect — one Bun `1.3.x` job with MinIO + the env that un-skips S3-compat). No
  standalone workflow file was added.
- **CORS validator location:** the contract sketched
  `packages/storage/src/cors/validate-cors.ts`; the validator instead lives in
  `scripts/validate-cors.ts` with unit tests in
  `scripts/__tests__/validate-cors.test.ts`. Behavior matches §4.
- **Test file naming:** adapter tests use `file-storage.conformance.test.ts`
  (Local, unconditional) and `file-storage.integration.test.ts` (S3 / S3-compat,
  `describe.skipIf`) co-located with the adapter, rather than the contract's
  `__tests__/<name>-file-storage.test.ts` paths. Equivalent gating, same suite.
- **Fixture reproducibility test:** `manifest.json` and the `FIXTURES` map exist
  and the generator is deterministic, but the contract's separate
  `fixtures.test.ts` re-hash oracle was **not** added — there is currently no
  automated test asserting committed-bytes sha256 == manifest. This is a contract
  extra, not a ROADMAP Success Criterion (SC#5 requires the set committed +
  consumed by conformance, both true). Logged for a quick follow-up.

## Adversarial review outcome

Adversarial review found **0 blockers**. The Pitfall-1 invariant (no raw storage
key in any typed `SignedUpload`/`SignedRead` result) is asserted directly in the
conformance suite (`expect("storageKey" in result).toBe(false)` + `"key"`), and
the Pitfall-14 local-in-prod ban is enforced by Phase 24's `validateStorageEnv`
(D-14) with an exact-message test in `env.test.ts`. The one non-blocking item is
the missing fixture-hash test noted above.
