Output: One new workflow file + one `.gitignore` edit + one checkpoint for the developer to configure Sentry/GitHub secrets and verify post-deploy.

    (a) Create `.github/workflows/release.yml`. Follow RESEARCH Example 5 structure. No PR jobs, no test jobs — tag-push only:

    ```yaml
    name: Release — upload source maps

    on:
      push:
        tags:
          - 'v*.*.*'

    jobs:
      upload-sourcemaps:
        name: Build + upload source maps to Sentry
        runs-on: ubuntu-latest
        # Required GitHub repo secrets — see Task 2 checkpoint:
        #   SENTRY_AUTH_TOKEN, SENTRY_ORG, SENTRY_PROJECT
        env:
          SENTRY_AUTH_TOKEN: ${{ secrets.SENTRY_AUTH_TOKEN }}
          SENTRY_ORG: ${{ secrets.SENTRY_ORG }}
          SENTRY_PROJECT: ${{ secrets.SENTRY_PROJECT }}
        steps:
          - name: Checkout repository
            uses: actions/checkout@v4
            with:
              fetch-depth: 0   # Required for git rev-parse --short HEAD

          - name: Setup Bun
            uses: oven-sh/setup-bun@v2
            with:
              bun-version: latest

          - name: Compute RELEASE (short git SHA)
            id: rel
            run: echo "release=$(git rev-parse --short HEAD)" >> "$GITHUB_OUTPUT"

          - name: Install dependencies
            run: bun install --frozen-lockfile

          - name: Build apps/api (+ worker bundle) with external source maps
            env:
              RELEASE: ${{ steps.rel.outputs.release }}
            # --sourcemap=external emits .map file with Debug ID and NO sourceMappingURL comment (Pitfall 5).
            # --define bakes $RELEASE into the bundle at build time (Pitfall 6 — single-source discipline).
            run: |
              bun build apps/api/src/index.ts  \
                --outdir apps/api/dist         \
                --target bun                   \
                --sourcemap=external           \
                --define "process.env.RELEASE=\"$RELEASE\""
              bun build apps/api/src/worker.ts \
                --outdir apps/api/dist/worker  \
                --target bun                   \
                --sourcemap=external           \
                --define "process.env.RELEASE=\"$RELEASE\""

          - name: Build apps/admin (Vite)
            env:
              VITE_RELEASE: ${{ steps.rel.outputs.release }}
            run: bun --cwd apps/admin run build

          - name: Build apps/web (Next.js — server-side source maps only)
            # productionBrowserSourceMaps stays FALSE (Pitfall 5). Next.js emits
            # server source maps into .next/server/* by default; those are the
            # ones we upload to Sentry. The browser bundle gets NO .map files
            # shipped to users.
            run: bun --cwd apps/web run build

          - name: Upload source maps to Sentry (all four apps)
            env:
              RELEASE: ${{ steps.rel.outputs.release }}
            # Debug ID variant — sentry-cli injects IDs into bundles + maps, then
            # uploads the JS/map pairs keyed by ID (not filename).
            # Works identically against Sentry and GlitchTip (shared wire protocol).
            run: |
              for DIR in apps/api/dist apps/admin/dist apps/web/.next; do
                echo "=== Processing $DIR ==="
                if [ -d "$DIR" ]; then
                  bun x sentry-cli sourcemaps inject "$DIR"
                  bun x sentry-cli sourcemaps upload \
                    --release="$RELEASE"            \
                    --org="$SENTRY_ORG"             \
                    --project="$SENTRY_PROJECT"     \
                    "$DIR"
                else
                  echo "WARNING: $DIR not found — build step did not produce output. Skipping."
                fi
              done

          - name: Summary
            env:
              RELEASE: ${{ steps.rel.outputs.release }}
            run: |
              echo "RELEASE=$RELEASE uploaded for apps: api, admin, web"
              echo "Runtime Sentry.init must read env.RELEASE=$RELEASE to match the uploaded maps."
    ```

    - Shell `if [ -d "$DIR" ]` guard in case a build step fails — workflow continues past that app rather than exiting

    2. Open the GitHub Actions tab. Watch the `Release — upload source maps` workflow run.
    3. Confirm all steps pass. If the "Upload source maps to Sentry" step fails with a 401, the `SENTRY_AUTH_TOKEN` scopes are insufficient — re-create it with `project:releases` AND `project:write`.

    **Part C — Deploy the built API AND trigger a deliberate failure** (~10 minutes; Success Criterion #4):

    1. Deploy the `apps/api/dist/index.js` build (produced by the workflow) to your staging environment, with `SENTRY_DSN`, `ERROR_TRACKER=sentry`, and `RELEASE=<short-sha>` set in the runtime env (the SAME short SHA used by the workflow — check the workflow's "Compute RELEASE" step output).
    2. Add a temporary endpoint that deliberately throws, e.g. `app.get("/boom", () => { throw new Error("Phase 18 release test"); });` — deploy this too.
    3. Hit the `/boom` endpoint once.
    4. Open Sentry/GlitchTip. Find the issue "Phase 18 release test".
    5. Open the Stack Trace tab. Every frame should show real source file paths + line numbers (`apps/api/src/index.ts:142`, for example — not `index-abc.js:1:2432`).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Sentry API | `sentry-cli` authenticates with `SENTRY_AUTH_TOKEN` to upload maps |
| GitHub repo → secrets | `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, `SENTRY_PROJECT` live in repo Secrets (masked in logs) |
| Tag push → workflow trigger | Only tags matching `v*.*.*` trigger the workflow |
| Build output → public assets | `.map` files must NOT be served alongside `.js` files (Pitfall 5) |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-18-42 | Information disclosure | `SENTRY_AUTH_TOKEN` leaked via workflow log echo | mitigate | Secrets are auto-masked in GitHub Actions logs (platform guarantee). Task 1 uses `env:` block, NOT `run: echo`. No `set -x` in run blocks. |

Fork PRs cannot trigger this workflow. D-16 narrow scope. |
| T-18-44 | Information disclosure | Source maps published to public CDN (Pitfall 5 — reverses minification, exposes source) | mitigate | `bun build --sourcemap=external` emits Debug ID without sourceMappingURL comment. `productionBrowserSourceMaps` not enabled in next.config. Task 2 Part D asserts public `.map` fetch returns 404. |

Pitfall 6 discipline. Task 2 Part C verifies the demangled trace. |
