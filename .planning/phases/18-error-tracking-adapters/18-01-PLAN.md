## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Information disclosure | `env.SENTRY_DSN` / `GLITCHTIP_DSN` values | mitigate | Zod `.url()` rejects non-URL strings; `createEnv` from @t3-oss/env-core prevents shell-injection into validators; DSN value never logged — only the env var NAME appears in thrown error messages (verified by Task 2 acceptance criteria) |
| T-18-02 | Tampering | Validator permissiveness — misconfigured env lets process boot with stale/wrong DSN | mitigate | Crash-hard discipline: `ERROR_TRACKER=sentry` without `SENTRY_DSN` throws in non-test env (Task 2). D-06 prohibits auto-detection from DSN presence; explicit `ERROR_TRACKER` selection is the only path. |
| T-18-03 | Denial of service | Boot-time throw loops the process under crash-restart supervisors | accept | Documented behavior; operators receive deterministic error with env var name. Alternative (silent noop-fallback) would leak errors to no backend — worse failure mode. |
| T-18-04 | Elevation of privilege | `OBS_PII_DENY_EXTRA_KEYS` as string allows extending scrubber denylist | mitigate | Plan 02 treats this as ADDITIVE only — the default hard-coded denylist is never REMOVED by env. The env value only appends keys. Reviewed in plan 02 scrub-pii.ts tests. |
| T-18-05 | Spoofing | Supply-chain risk — npm registry returns malicious `@sentry/bun` or `@sentry/cli` | accept | Pin to known-good versions (`^10.49.0`, `^3.4.0`); `bun install --frozen-lockfile` in CI. Full package-provenance verification is out of scope for this phase (broader monorepo concern). |
</threat_model>
