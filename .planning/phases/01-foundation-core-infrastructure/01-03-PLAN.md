**packages/db/src/helpers/scoped-db.ts** (per D-10):
- Export `ScopedDb` type for use in HandlerContext.
- Export `scopedDb(db: DbInstance, tenantId: string)` function that returns an object with:
  - `select<T extends PgTable & { tenantId: any }>(table: T)` -- returns `db.select().from(table).where(eq(table.tenantId, tenantId))`. This returns a Drizzle query builder that the caller can further chain (.limit, .orderBy, etc.).
  - `insert<T extends PgTable & { tenantId: any }>(table: T)` -- returns an object with `values(data)` that calls `db.insert(table).values({ ...data, tenantId } as any).returning()`. The tenantId is auto-injected so callers cannot forget it.
  - `update<T extends PgTable & { tenantId: any }>(table: T)` -- returns `db.update(table).set({}).where(eq(table.tenantId, tenantId))`. Note: the caller chains `.set(data)` after. Actually, Drizzle's update API is `db.update(table).set(data).where(...)`. So the wrapper should return something that accepts `.set(data)` and auto-applies the where clause. Implementation approach: return `{ set: (data: any) => db.update(table).set(data).where(eq(table.tenantId, tenantId)) }`.
  - `delete<T extends PgTable & { tenantId: any }>(table: T)` -- returns `db.delete(table).where(eq(table.tenantId, tenantId))`. Caller can chain additional `.where()` conditions.
  - `tenantId` property -- exposed so handlers can reference the current tenant ID.
  - `raw` property -- the underlying Drizzle instance, for complex queries where the caller must manually include tenant filtering. Marked with a JSDoc warning: "Use with caution -- no automatic tenant filtering."

Import `eq` from `drizzle-orm`, `PgTable` from `drizzle-orm/pg-core`.

  - Other errors: 500 with `{ success: false, error: 'INTERNAL_ERROR' }` (do not expose error details in production).
- Log all errors via pino logger.
- Export as `errorMiddleware`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP Header -> Tenant Context | x-tenant-id header is untrusted (Phase 1 placeholder; Phase 2 replaces with session-derived tenant) |
| Tenant Context -> Database | scopedDb must enforce tenant filtering on every query |
| Module Route -> Handler | Route handlers must pass the middleware-injected context, not construct their own |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-08 | Information Disclosure | packages/db/src/helpers/scoped-db.ts | mitigate | scopedDb auto-applies WHERE tenant_id = ? on all select/update/delete and auto-injects tenant_id on insert. Cross-tenant access requires explicit use of unscopedDb() or the .raw escape hatch (which is documented as dangerous). |
| T-01-09 | Tampering | apps/api/src/core/middleware/tenant.ts | accept (Phase 1 only) | Phase 1 extracts tenantId from x-tenant-id header which is client-controlled. This is acceptable for Phase 1 development only. Phase 2 MUST replace this with session-derived tenant from better-auth. A TODO comment marks this. |
| T-01-10 | Spoofing | x-tenant-id header | accept (Phase 1 only) | Same as T-01-09. Client can spoof tenant in Phase 1. Phase 2 mitigates by deriving tenant from authenticated session. |
| T-01-11 | Information Disclosure | apps/api/src/core/middleware/error.ts | mitigate | Error middleware returns generic INTERNAL_ERROR in production, never exposes stack traces or internal error details. Detailed errors logged server-side via pino. |
| T-01-12 | Elevation of Privilege | apps/api/src/worker.ts | mitigate | Worker loads modules with role='worker', which skips route attachment. Worker cannot serve HTTP requests even if misconfigured. |
</threat_model>
