# Phase 11 Plan 02: InviteDialog Form Primitive Refactor Summary

**InviteDialog email + role fields rewired through @baseworks/ui Form primitives, with a schema-factory pattern that puts pre-translated zod error messages directly into errors.email.message so FormMessage announces them via role="alert" without conditional JSX.**

## Accomplishments

- Closed GAP-3 from the v1.1 milestone audit (raw `<p className="text-sm text-destructive">` error paragraphs eliminated)
- Closed A11Y-04 + A11Y-05 by routing email + role fields through FormField/FormItem/FormLabel/FormControl/FormMessage — `role="alert"`, `aria-describedby`, and `aria-invalid` are now wired automatically via FormControl + FormMessage
- Schema factory `buildEmailSchema(t)` carries translated error messages directly on each zod failure path (`emailRequired` for empty, `emailInvalid` for bad format) — eliminated the `errors.email.type === "too_small"` branching from the JSX
- Net typecheck error count REDUCED by 2: my refactor fixed the original Resolver-mismatch and SubmitHandler-mismatch errors that the legacy JSX was hiding via `form.register` + property access

## Decisions Made

- **Refactored both email AND role fields** (not just email) to FormField. The plan called for this; the audit's "bypasses Form primitives" language implicates the whole dialog.
- **Schema factory pattern** instead of inlining branching into FormMessage's render prop. Cleaner: FormMessage renders `String(errors.email.message)` unconditionally; the i18n logic lives at schema construction time.
- **Resolver cast via `as unknown as`** — the conditional `isLinkMode ? linkSchema : emailSchema` produces a union type that does not unify with `Resolver<EmailFormValues>`. The runtime is correct because the `email` field is unmounted in link mode (so linkSchema validates only mounted fields). Documented inline.

**2. [Rule 1 - Bug] Added Resolver type cast for FormField type-pinning**
- **Found during:** Task 2 (typecheck after JSX refactor)
- **Issue:** Adding `FormField control={form.control}` introduced two new TS2322 errors. The conditional resolver `isLinkMode ? linkSchema : emailSchema` produced a union `Resolver<linkSchema> | Resolver<emailSchema>` that TypeScript widened to `Control<EmailFormValues, any, TFieldValues>`, which `FormField` could not unify with `Control<EmailFormValues, any, EmailFormValues>`.
- **Fix:** Pinned the resolver type via `as unknown as Resolver<EmailFormValues>`. Added `import { type Resolver } from "react-hook-form"`. Documented the runtime correctness reasoning inline (email field unmounted in link mode → linkSchema validates only mounted fields).
- **Files modified:** apps/web/components/invite-dialog.tsx

After my refactor: 3 errors remain (invitations API, toast Date typing, Switch checked any) — all PRE-EXISTING and unrelated to a11y. The Resolver mismatch and SubmitHandler mismatch were collateral fixes from the refactor.
