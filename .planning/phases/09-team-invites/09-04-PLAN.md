**Signup page (`apps/web/app/(auth)/signup/page.tsx`) -- D-08 AUTO-ACCEPT:**

This is CRITICAL for D-08. After successful signup with an invite token, the user must be auto-accepted into the organization WITHOUT being redirected back to /invite/[token] to manually click Accept. The flow is: signup -> auto-accept -> set active org -> redirect to /dashboard.

1. Read both `invite` and `email` query params from URL.
2. If `email` param exists, pre-fill the email field in the signup form. This means setting the default value of the email input or calling `form.setValue("email", emailParam)` if using react-hook-form.
3. **After successful signup, if `invite` param exists, perform auto-accept:**

```typescript
const searchParams = useSearchParams();
const inviteToken = searchParams.get("invite");
const inviteEmail = searchParams.get("email");

// Pre-fill email in form default values:
const form = useForm({
  defaultValues: {
    email: inviteEmail || "",
    // ... other fields
  },
});

// In signup success handler:
if (inviteToken) {
  try {
    // Step 1: Auto-accept the invitation (D-08)
    await auth.organization.acceptInvitation({ invitationId: inviteToken });

    // Step 2: Fetch invitation to get organizationId for setActive
    const response = await fetch(`/api/invitations/${inviteToken}`);
    const invitation = await response.json();

    // Step 3: Set the org as active tenant (Pitfall 3)
    if (invitation?.organizationId) {
      await auth.organization.setActive({ organizationId: invitation.organizationId });
    }

    // Step 4: Go directly to dashboard (D-09)
    router.push("/dashboard");
  } catch (error) {
    // If auto-accept fails (e.g., invitation was cancelled meanwhile),
    // still redirect to dashboard -- user has an account now
    console.error("[SIGNUP] Auto-accept failed:", error);
    router.push("/dashboard");
  }
} else {
  router.push("/dashboard");
}
```

**IMPORTANT implementation notes:**

- The auto-accept MUST happen in the signup success handler, NOT via redirect to /invite/[token]. D-08 says "auto-accept on signup completion" -- the user should never see the invite page after creating their account.
- The `auth.organization.acceptInvitation()` call requires the user to be authenticated, which they are immediately after signup (better-auth creates a session on signup).
- If the invitation was for a specific email and the user signs up with a different email, better-auth may reject the acceptance. This is expected behavior -- the user still gets their account.
- Read the existing login and signup page files FIRST to understand their current structure, form setup, and redirect patterns. Do not break existing functionality. The invite handling is additive.
- If login/signup currently redirects via `router.replace()`, use the same method for the invite redirect to avoid back-button issues.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public invite page -> /api/invitations/:id | Unauthenticated read of invitation details |
| Invite page -> auth.organization.acceptInvitation | Requires authenticated session |
| Query params (?invite=) | Token passed through login/signup redirects |
| Signup auto-accept -> auth.organization.acceptInvitation | Requires fresh session from signup |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-11 | Information Disclosure | /invite/[token] page | mitigate | Public page only shows org name, inviter name, and role. No internal org details, member lists, or configuration exposed. Invitation IDs are opaque random strings. |
| T-09-12 | Spoofing | Login redirect with invite param | mitigate | Invite param is validated server-side when accept is called -- invalid tokens are rejected by better-auth. The param only controls redirect destination within the app (no open redirect). Validate that redirect goes to `/invite/` path only. |
| T-09-13 | Elevation of Privilege | Accept invitation as different user | mitigate | better-auth validates that the accepting user's email matches the invitation email (for email invites). For link invites, any authenticated user can accept (by design per D-10). |
| T-09-14 | Repudiation | Invite acceptance | accept | Invitation status tracked in database by better-auth. Events emitted for audit. Low risk for starter kit. |
| T-09-15 | Elevation of Privilege | Signup auto-accept with mismatched email | mitigate | If user signs up with different email than invitation target, better-auth rejects the acceptInvitation call. Catch block handles gracefully -- user gets account but not org membership. |
</threat_model>
