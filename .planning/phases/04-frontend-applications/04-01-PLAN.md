**Per D-02: Fix `as any` casts in `apps/api/src/index.ts`.**

The goal is to preserve the Elysia method chain so `export type App = typeof app` captures all route types for Eden Treaty inference.

Strategy: Instead of `app.use(authRoutes as any)` and `registry.attachRoutes(app as any)`, compose routes using Elysia's `.use()` method chaining directly. The registry's `attachRoutes` method must return the routes plugin(s) rather than mutating the app.

1. **Refactor `registry.ts`**: Change `attachRoutes(app)` to `getModuleRoutes(): Elysia plugin`. The method should return a single Elysia instance that `.use()`-chains all non-auth module route plugins. This way the app can do `app.use(registry.getModuleRoutes())` in the chain. Keep `attachRoutes` as a deprecated alias if needed for backward compat.

2. **Refactor `apps/api/src/index.ts`**: Build the app via a single chained expression:
   ```typescript
   const app = new Elysia()
     .use(errorMiddleware)
     .use(cors({ credentials: true, origin: [env.WEB_URL, env.ADMIN_URL].filter(Boolean) }))
     .use(swagger())
     .get("/health", () => ({ status: "ok", modules: registry.getLoadedNames() }))
     .use(authRoutes)  // No as any -- auth routes plugin must be typed
     .use(tenantMiddleware)
     .derive({ as: "scoped" }, (ctx) => { /* handlerCtx */ })
     .use(billingApiRoutes)  // New billing HTTP routes
     .use(adminRoutes)       // New admin routes
     .group("/api", (group) => group.use(requireRole("owner")).delete("/tenant", ...))
     .use(registry.getModuleRoutes());  // Non-auth, non-billing module routes

   export type App = typeof app;
   ```

   If full type chain proves impossible due to registry generics, use a typed route aggregator pattern: extract all routes as Elysia plugin instances and `.use()` them in a chain. The key constraint: NO `as any` anywhere in the chain before `export type App`.

   Add env vars for CORS: Add `WEB_URL` and `ADMIN_URL` optional string env vars to `packages/config/src/env.ts` (for CORS origin list and better-auth trustedOrigins). Default WEB_URL to `http://localhost:3000` and ADMIN_URL to `http://localhost:5173`.

**Per D-03: Add billing HTTP route handlers to `packages/modules/billing/src/routes.ts`.**

The file currently only has the webhook POST route. Add HTTP routes wrapping each CQRS command and query. All these routes sit under the existing `/api/billing` prefix and require tenant context (they run after tenant middleware).

Add these routes to the existing `billingRoutes` Elysia instance:

- `POST /api/billing/checkout` -- calls `createCheckoutSession.handler({ priceId, successUrl, cancelUrl }, ctx.handlerCtx)`. Body: `{ priceId: string, successUrl: string, cancelUrl: string }`. Returns `{ success: true, data: { url: string } }` or error.
- `POST /api/billing/cancel` -- calls `cancelSubscription.handler({}, ctx.handlerCtx)`. No body. Returns `{ success: true }` or error.
- `POST /api/billing/change` -- calls `changeSubscription.handler({ newPriceId }, ctx.handlerCtx)`. Body: `{ newPriceId: string }`. Returns `{ success: true }` or error.
- `POST /api/billing/one-time` -- calls `createOneTimePayment.handler({ priceId, quantity, successUrl, cancelUrl }, ctx.handlerCtx)`. Body matches command input. Returns `{ success: true, data: { url: string } }` or error.
- `POST /api/billing/portal` -- calls `createPortalSession.handler({ returnUrl }, ctx.handlerCtx)`. Body: `{ returnUrl: string }`. Returns `{ success: true, data: { url: string } }` or error.
- `POST /api/billing/usage` -- calls `recordUsage.handler({ featureKey, quantity }, ctx.handlerCtx)`. Body matches command input. Returns `{ success: true }` or error.
- `GET /api/billing/subscription` -- calls `getSubscriptionStatus.handler({}, ctx.handlerCtx)`. Returns subscription status data.
- `GET /api/billing/history` -- calls `getBillingHistory.handler({ limit?, offset? }, ctx.handlerCtx)`. Query params: `limit` (default 20), `offset` (default 0). Returns billing history array.

Each route handler follows the result pattern: check `result.success`, if false set status 400 and return error. Import the command/query handlers directly from their files.

**Per D-16: Add admin API routes in `apps/api/src/routes/admin.ts`.**

Create a new file `apps/api/src/routes/admin.ts` exporting an Elysia plugin with prefix `/api/admin`. These routes use raw `db` (not scopedDb) for cross-tenant queries. Protect with `requireRole("owner")`.

Routes:
- `GET /api/admin/tenants` -- lists all organizations from the `organization` table (better-auth). Returns `{ data: Array<{ id, name, slug, createdAt, metadata }> }`. Support `?limit=20&offset=0&search=` query params.
- `GET /api/admin/tenants/:id` -- get single organization by ID with member count.
- `PATCH /api/admin/tenants/:id` -- update organization metadata (e.g., deactivate). Body: `{ metadata: Record<string, any> }`.
- `GET /api/admin/users` -- lists all users from the `user` table. Returns `{ data: Array<{ id, name, email, image, createdAt }> }`. Support `?limit=20&offset=0&search=`.
- `GET /api/admin/users/:id` -- get single user with their organization memberships.
- `PATCH /api/admin/users/:id` -- update user (ban/unban). Body: `{ banned: boolean, banReason?: string }`.
- `POST /api/admin/users/:id/impersonate` -- create impersonation session. Implementation: use better-auth admin API if available, otherwise create a custom session via `auth.api.createSession({ userId: targetId })` and return the session token.
- `GET /api/admin/billing/overview` -- aggregate billing stats across all tenants: total subscribers, MRR estimate (count of active subscriptions), subscription distribution by plan. Queries `billing_customers` table.
- `GET /api/admin/system/health` -- returns BullMQ queue depths (query each registered queue), Redis memory usage (via ioredis `info memory`), and API uptime. Import queue references from billing module or registry.

Import `createDb` from `@baseworks/db`, `requireRole` from `@baseworks/module-auth`. The admin routes use a fresh db instance (not tenant-scoped). Use Drizzle's `organization`, `user`, `member` tables from the auth schema and `billingCustomers` from billing schema.
