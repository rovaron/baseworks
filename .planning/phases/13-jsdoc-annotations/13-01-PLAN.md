**Step 1: Create docs/jsdoc-style-guide.md**

Create the file at `docs/jsdoc-style-guide.md`. This is the prescriptive style guide (per JSDOC-06). Structure it with these sections:

1. **Purpose** -- one paragraph: "This guide standardizes JSDoc across the Baseworks codebase..."
2. **General Rules** -- Biome compatibility (100-char line width, 2-space indent in comment blocks, no `@type` in .ts files, `@warning` is allowed), always include `@param` and `@returns` per D-03, technical-precise tone per D-02
3. **Tag Ordering** -- exact order from 13-PATTERNS.md section 4: description, @param, @returns, @throws, @example, @see, Per X-YY references
4. **Templates by Export Type** -- copy the 6 templates from 13-PATTERNS.md section 2 verbatim (defineCommand/defineQuery handlers, plain functions, interfaces/types, classes, Elysia plugins, schema tables)
5. **Good vs Bad Examples** -- at least 3 pairs:
   - Bad: `/** Creates a tenant. */` vs Good: full block with @param/@returns describing validated input, Result return, events emitted
   - Bad: `/** @param name - name */` (restating type) vs Good: `/** @param name - Display name for the organization, 2-50 characters */` (adding constraint info)
   - Bad: no JSDoc on exported type vs Good: interface block describing domain purpose and constraints
6. **@example Guidelines** -- per D-05/D-06: minimal 3-5 line runnable snippets, no setup boilerplate, target list of 10-15 functions
7. **When to Skip JSDoc** -- barrel re-export files (index.ts), trivially self-documenting accessors (getFoo), test files, standard shadcn components
