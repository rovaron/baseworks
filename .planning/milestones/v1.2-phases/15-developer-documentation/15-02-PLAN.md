Step 2b -- Create `packages/modules/example/src/hooks/on-example-created.ts` (mirrors `packages/modules/billing/src/hooks/on-tenant-created.ts` exactly):

```typescript
import { env } from "@baseworks/config";
import { createQueue } from "@baseworks/queue";
import type { Queue } from "bullmq";

/**
 * Auto-enqueue a `example:process-followup` job when an example record
 * is created.
 *
 * Per D-05 and Plan 15-02: The example module demonstrates all four
 * module surfaces (command, query, event, BullMQ job). The command
 * emits `example.created`; this hook subscribes to that event and
 * enqueues the follow-up job. Mirrors the canonical event-bus-hook
 * pattern in `packages/modules/billing/src/hooks/on-tenant-created.ts`.
 *
 * Registered as a listener on `example.created` via the TypedEventBus
 * provided by the module registry. Called once at API startup from
 * `apps/api/src/index.ts`.
 */

interface ExampleCreatedEvent {
  id: string;
  tenantId: string;
}

/**
 * Lazy-initialized BullMQ queue for example:process-followup.
 * Only created if REDIS_URL is available; falls back to console.log
 * when Redis is not configured (dev/test). Mirrors the `getEmailQueue`
 * pattern in `packages/modules/auth/src/auth.ts:15-21`.
 */
let followupQueue: Queue | null = null;
function getFollowupQueue(): Queue | null {
  if (!followupQueue && env.REDIS_URL) {
    followupQueue = createQueue("example:process-followup", env.REDIS_URL);
  }
  return followupQueue;
}

/**
 * Register example-module hooks on the event bus.
 *
 * Attaches an `example.created` listener that enqueues the follow-up
 * job onto the `example:process-followup` BullMQ queue. On error,
 * logs and does not rethrow -- failure of the hook MUST NOT crash
 * the originating command.
 *
 * @param eventBus - The TypedEventBus instance from the module registry
 * @returns void
 */
export function registerExampleHooks(eventBus: {
  on: (event: string, handler: (data: any) => Promise<void>) => void;
}): void {
  eventBus.on("example.created", async (data: unknown) => {
    const { id, tenantId } = data as ExampleCreatedEvent;

    try {
      const queue = getFollowupQueue();
      if (!queue) {
        // biome-ignore lint/suspicious/noConsole: graceful dev-without-Redis fallback
        console.log(
          `[example] Skipping process-followup enqueue for ${id} (REDIS_URL not set)`,
        );
        return;
      }

      await queue.add("example:process-followup", { exampleId: id, tenantId });
    } catch (err) {
      // biome-ignore lint/suspicious/noConsole: error path of a best-effort hook
      console.error(
        `[example] Failed to enqueue process-followup for ${id}:`,
        err,
      );
    }
  });
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API process → TypedEventBus | `ctx.emit("example.created", ...)` crosses from the command handler into the in-process event bus. The bus is in-memory; no serialization. |
| Event-bus listener → BullMQ/Redis | `registerExampleHooks` calls `queue.add(...)`; the payload is serialized and persisted in Redis until the worker consumes it. |
| Worker process → downstream (logs) | `processFollowup` reads job data and writes to stdout; worker runs with its own Redis connection, separate from the API request context. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-03 | Tampering | Job payload crosses the app ↔ Redis boundary; a compromised Redis could inject crafted payloads into the worker. | accept | Redis is treated as a trust-internal service behind the VPC boundary (per existing `REDIS_URL` usage in `packages/queue/src/index.ts`). The demo handler logs only scalar fields from the typed payload and performs no database writes, so a crafted payload cannot escalate. Any future handler that updates the database MUST use `tenantId` from the job data in its WHERE clause and re-validate before mutating state. |
| T-15-04 | Information Disclosure | `processFollowup` logs `tenantId` and `exampleId` to stdout. | accept | `exampleId` is an internal row id and `tenantId` is already present in every request log via `requestTraceMiddleware`; no PII is logged. Logging is required for operator debuggability of the worker. |
| T-15-05 | Denial of Service | Crafted input could cause `processFollowup` to throw repeatedly, consuming BullMQ retry slots. | mitigate | BullMQ defaults (3 attempts with exponential backoff; 7-day failure retention) apply automatically via `createQueue`. The demo handler has no external calls and cannot hang. No further mitigation required for the tutorial-sized handler. |
| T-15-06 | Elevation of Privilege | Handler accesses tenant-scoped data without going through `scopedDb`. | mitigate | The demo handler performs NO database writes or reads -- it only logs. The "Add a Module" tutorial (Plan 04) will explicitly document that any worker handler performing tenant-scoped DB work MUST build its own `scopedDb` from `tenantId` in the job payload (pattern: `packages/modules/billing/src/hooks/on-tenant-created.ts::registerBillingHooks`). |
| T-15-21 | Denial of Service | A flood of `example.created` events could saturate Redis via unbounded enqueues. | accept | In-process bus; enqueue rate is bounded by the same `createExample` rate that is already rate-limited upstream. BullMQ retains only the most recent 3 days of completed jobs, bounding storage. |
</threat_model>
