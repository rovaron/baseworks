# Milestone v1.3: Observability & Operations

**Status:** 🚧 IN PROGRESS (started 2026-04-21)
**Phases:** 17-23
**Total Plans:** TBD (derived per-phase)
**Goal:** Ship production-grade observability and ops tooling so operators running Baseworks can detect, diagnose, and resolve incidents without SSHing into boxes or grep-ing logs.
**Architecture:** Port + adapters (mirroring the billing `PaymentProvider` pattern). Fork users select backends via env vars; Noop adapters ship as defaults so the system runs with zero external dependencies.
**Coverage:** 28/28 v1.3 requirements mapped (no orphans, no duplicates).

## Overview

v1.3 adds a complete observability layer (errors, metrics, traces, structured context logging) and ops tooling (bull-board, health dashboard, runbooks) on top of the v1.0–v1.2 baseline. Every instrumentation point is applied as an external wrapper around existing core primitives (CqrsBus, TypedEventBus, Drizzle client, BullMQ Queue/Worker) — zero edits to business handlers. A single `AsyncLocalStorage<ObservabilityContext>` carries `{requestId, traceId, spanId, tenantId, userId}` through every request, event, and job. Pino stays the canonical log surface; ErrorTracker adapters tee from it rather than replace it.

Three Bun-specific constraints drive the phase structure:

1. OTEL SDK must be bootstrapped as line-1 of every entrypoint (Bun does not honour `NODE_OPTIONS=--require`).
2. `AsyncLocalStorage#enterWith` is banned — only `.run()` is permitted (cross-tenant context bleed under load).
3. `@sentry/bun` only; `@sentry/profiling-node` is broken under Bun.

## Phases

- [ ] **Phase 17: Observability Ports & OTEL Bootstrap** — Typed ports (Tracer / Metrics / ErrorTracker) with Noop adapters and a Bun-safe OTEL SDK bootstrap behind startup env validation
- [ ] **Phase 18: Error Tracking Adapters** — Sentry + GlitchTip + Pino-sink adapters with PII scrubbing, context enrichment, and source-map uploads on release
- [ ] **Phase 19: Context, Logging & HTTP/CQRS Tracing** — Single ALS context carrier, pino mixin, Elysia middleware, and external CqrsBus/EventBus tracer wrappers
- [ ] **Phase 20: BullMQ Trace Propagation** — W3C traceparent injected at enqueue, extracted at worker pickup, producing a single trace tree across API → job
- [ ] **Phase 21: OTEL Adapters + Grafana Observability Stack** — OTLP exporters with cardinality guardrails, `docker-compose.observability.yml`, and 4 provisioned Grafana dashboards
- [ ] **Phase 22: Admin Ops Tooling** — bull-board behind RBAC, admin iframe, health dashboard upgrade with queue depth, worker heartbeat, DB lag, recent errors
- [ ] **Phase 23: Runbooks, Alert Templates & Observability Docs** — 8–10 incident runbooks, Grafana alert YAML + Sentry templates with `runbook_url` annotations, and observability concepts doc

## Phase Details

### Phase 17: Observability Ports & OTEL Bootstrap

**Goal**: Operator can start the API and worker with typed observability ports wired and OTEL bootstrapped first-in-entrypoint, with zero external dependencies when defaults are kept.
**Depends on**: Phase 16 (v1.2 complete)
**Requirements**: OBS-01, OBS-02, OBS-03, OBS-04
**Success Criteria** (what must be TRUE):
  1. Operator sees `packages/observability` expose `ErrorTracker`, `MetricsProvider`, and `Tracer` port types plus Noop adapters wired into API and worker at startup via a factory selected by `@t3-oss/env-core` config
  2. Operator running with default env (no DSNs, `TRACER=noop`, `METRICS_PROVIDER=noop`) sees the API and worker start cleanly with no external dependency and no spurious OTEL traffic
  3. Operator sees `import "./telemetry"` as line 1 of `apps/api/src/index.ts` and `apps/api/src/worker.ts`, with a startup self-test span logged as `otel-selftest: ok`
  4. Operator sees a Bun smoke-test in CI that boots the telemetry module and asserts every enabled auto-instrumentation (HTTP, pino, ioredis) loads without crashing — `fs`/`dns`/`net` instrumentations are explicitly disabled
  5. Operator sees `validateObservabilityEnv()` fail fast at startup when required DSNs/endpoints are missing for the selected adapter

### Phase 18: Error Tracking Adapters

**Goal**: Operator can capture errors with rich context and zero PII leakage, swapping Sentry ↔ GlitchTip ↔ Pino-sink via DSN, with prod stack traces readable by release.
**Depends on**: Phase 17
**Requirements**: ERR-01, ERR-02, ERR-03, ERR-04, EXT-01
**Success Criteria** (what must be TRUE):
  1. Operator setting `SENTRY_DSN` sees uncaught exceptions, CQRS handler errors, and BullMQ job failures captured in Sentry with release + environment tags; pointing the same DSN at GlitchTip produces identical events (adapter-conformance test proves parity)
  2. Operator with no DSN configured sees the Pino-sink fallback adapter log errors at ERROR level with full structured context and zero external calls
  3. Operator sees every captured error enriched with `tenant_id`, `user_id`, `request_id`, and command/query name, and sees a conformance test feed known-PII fixtures (passwords, tokens, emails, CPF/CNPJ, Stripe/Pagar.me webhook bodies) and assert they are redacted before send
  4. Developer pushing a release git tag sees a GitHub Actions step upload source maps to Sentry/GlitchTip for the API, worker, and admin, and sees a demangled stack trace for a deliberately-failing endpoint post-deploy
**Plans**: TBD

### Phase 19: Context, Logging & HTTP/CQRS Tracing

**Goal**: Operator sees every log line and span carry `trace_id`, `span_id`, `requestId`, and `tenantId` automatically, with no handler-file edits and no cross-tenant context bleed under load.
**Depends on**: Phase 17
**Requirements**: CTX-01, CTX-02, CTX-03, TRC-01, TRC-02
**Success Criteria** (what must be TRUE):
  1. Operator sees a single `AsyncLocalStorage<ObservabilityContext>` populated by an Elysia `observabilityMiddleware` that reads inbound W3C `traceparent` or starts a new trace and derives tenant/user from the existing tenant middleware
  2. Operator sees every existing pino log line (across every module) automatically include `trace_id`, `span_id`, `requestId`, and `tenantId` via a logger mixin, with zero call-site changes across any handler
  3. Operator sees every HTTP request produce a server span named by method + Elysia route template + status code, accepting inbound `traceparent` and emitting it on outbound responses
  4. Operator sees every CQRS command/query dispatch and every event publish produce a child span with correlation attributes, emitted by an external `wrapCqrsBus` / `wrapEventBus` wrapper without any edit to `core/cqrs.ts`, `core/event-bus.ts`, or any handler file
  5. Operator running a 100 RPS concurrent-tenant load test sees every log line carry the correct `tenantId` for its request; a Biome/ESLint rule banning `AsyncLocalStorage#enterWith` fails CI when introduced
**Plans**: TBD

### Phase 20: BullMQ Trace Propagation

**Goal**: Operator debugging a failed background job sees a single distributed trace spanning API request → enqueue → worker pickup → handler in Tempo.
**Depends on**: Phase 19
**Requirements**: CTX-04, TRC-03
**Success Criteria** (what must be TRUE):
  1. Operator sees BullMQ `queue.add(...)` injected with W3C `traceparent` + `requestId` into `job.data._otel`, and sees workers reconstitute ALS context via `obsContext.run(...)` on job pickup
  2. Operator running an end-to-end test that triggers an API request which enqueues a job sees a single trace in Tempo where the worker's process span is a child of the API's enqueue span (same `traceId`)
  3. Operator sees enqueue instrumentation wired via `@appsignal/opentelemetry-instrumentation-bullmq` (or hand-rolled W3C propagator fallback) and gated by a Bun smoke test as a merge requirement
  4. Operator sees worker logs inherit the producer's `trace_id`, `requestId`, and `tenantId` from the job carrier — confirmed by grepping a sample log line from an enqueued-then-processed job
**Plans**: TBD

### Phase 21: OTEL Adapters + Grafana Observability Stack

**Goal**: Operator can launch a local observability stack with one command and see RED (routes + CQRS) and USE (DB/Redis/queue) metrics populating four pre-provisioned Grafana dashboards.
**Depends on**: Phase 20
**Requirements**: MET-01, MET-02, MET-03, DOC-01, DOC-02
**Success Criteria** (what must be TRUE):
  1. Operator setting `OTEL_ENABLED=true` and `OTEL_EXPORTER_OTLP_ENDPOINT=...` sees OTLP/HTTP+proto export of traces and metrics to an OTEL Collector; Noop remains the zero-overhead default
  2. Operator sees RED metrics (rate, errors, p50/p95/p99 duration) per Elysia route template and per CQRS command/query name, with a unit test asserting no `tenant_id`, `user_id`, or raw URL-path labels escape onto metrics (cardinality guardrail)
  3. Operator sees USE metrics for DB connection pool, Redis, and BullMQ queues (depth, active, failed/sec) — aggregate only, never per-tenant — enforced by OTEL Views plus collector transform/filter processors
  4. Developer can run `bun run observability:up` to launch `docker-compose.observability.yml` (OTEL Collector Contrib 0.127 + Tempo 2.10 + Loki 3.7 + Prometheus 3.10 + Grafana 12.4 remapped to port 3030) with per-service `mem_limit` and ≤24h dev retention; total footprint stays under ~2 GB RAM
  5. Operator logging into Grafana sees 4 pre-provisioned dashboards on first boot: API Overview (RED), Queue Health (BullMQ), DB+Redis (USE), and CQRS View (per-command RED)
**Plans**: TBD

### Phase 22: Admin Ops Tooling

**Goal**: Admin user can monitor jobs and system health from the Vite admin dashboard without leaving the app, with bull-board gated by RBAC and read-only by default.
**Depends on**: Phase 20, Phase 21
**Requirements**: OPS-01, OPS-02, OPS-03, OPS-04, EXT-02
**Success Criteria** (what must be TRUE):
  1. Operator sees `@bull-board/elysia` mounted at `/admin/bull-board` behind `requireRole("owner")` with read-only mode enabled by default via feature-flag env and admin-origin CSP; unauthenticated requests return 401, non-admin return 403, and static asset requests are also gated
  2. Admin user sees a "Job Monitor" entry in the Vite admin dashboard sidebar that renders bull-board as a same-origin iframe sharing the better-auth session cookie
  3. Admin user sees a `/health/detailed` endpoint and matching admin dashboard page showing queue depth with warn/critical thresholds, worker heartbeat freshness, DB lag, recent errors, and per-module status
  4. Module author sees a `HealthContributor` registered at module registration time; the central aggregator rolls up all contributions into an overall status surfaced by the admin health page
  5. Operator sees workers publishing heartbeat keys to Redis on a configurable interval so the health dashboard worker-heartbeat status reflects real state, not a mock

### Phase 23: Runbooks, Alert Templates & Observability Docs

**Goal**: Operator paged at 3am has a linked runbook for every alert and a short doc explaining how attributes, cardinality, and trace propagation work in this codebase.
**Depends on**: Phase 21, Phase 22
**Requirements**: DOC-03, DOC-04
**Success Criteria** (what must be TRUE):
  1. Operator sees 8–10 incident runbooks under `docs/runbooks/` covering DB down, Redis down, queue backing up, webhook failures, auth outage, OTEL exporter failing, bull-board inaccessible, high error rate, and slow checkout — each using a Trigger → Symptoms → Triage → Resolution → Escalation template
  2. Operator gets a pre-built Grafana alert rule YAML plus Sentry alert config templates importable into their tooling, with a `runbook_url` annotation on every rule linking to the matching `docs/runbooks/*.md` file
  3. Developer sees an observability concepts doc at `docs/observability/` covering the attributes glossary (which go on spans/logs vs metrics), the cardinality guide, and the trace-propagation flow (API → DB → enqueue → worker)
  4. Operator sees alerts designed with SLO-based burn-rate thresholds and `for: 5m` minimums so deploy rollouts and warmup periods do not fire, with runbook links living in-repo so CI fails if a `runbook_url` points to a missing file
