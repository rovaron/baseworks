<objective>
Build the module registry, CQRS command/query bus, in-process event bus, and an example module that proves the entire pattern works end-to-end.

Purpose: This plan creates the core architectural machinery. The module registry is the "load-bearing wall" of Baseworks -- every future module (auth, billing, etc.) plugs into it. The example module validates that the registry-to-CQRS-to-events pipeline works before real modules are built on it.

Output: A running Elysia server that dynamically loads the example module from config, dispatches commands/queries through the CQRS bus, and emits domain events via the event bus.
</objective>

**apps/api/src/core/event-bus.ts** (per D-06):
- Create `TypedEventBus` class wrapping Node.js `EventEmitter` (from `node:events`).
- Methods: `emit(event: string, data: unknown): void`, `on(event: string, handler: (data: any) => void | Promise<void>): void`, `off(event: string, handler: Function): void`.
- Use the `DomainEvents` interface from `@baseworks/shared` for type safety where possible, but keep a flexible string-based API since the events map is extended by modules.
- Handle async subscriber errors: wrap subscriber calls in try/catch, log errors via logger but do not rethrow (fire-and-forget per D-06).

**apps/api/src/core/cqrs.ts** (per D-05, D-08):
- Create `CqrsBus` class with:
  - `private commands = new Map<string, CommandHandler<any, any>>()`
  - `private queries = new Map<string, QueryHandler<any, any>>()`
  - `registerCommand(name: string, handler: CommandHandler<any, any>): void`
  - `registerQuery(name: string, handler: QueryHandler<any, any>): void`
  - `async execute<T>(command: string, input: unknown, ctx: HandlerContext): Promise<Result<T>>` -- looks up handler, returns `err('COMMAND_NOT_FOUND')` if missing, otherwise calls handler with input and ctx.
  - `async query<T>(queryName: string, input: unknown, ctx: HandlerContext): Promise<Result<T>>` -- same pattern for queries.
- Import `HandlerContext`, `Result`, `CommandHandler`, `QueryHandler`, `err` from `@baseworks/shared`.

**apps/api/src/core/registry.ts** (per D-01, D-02, D-03):
- Create `ModuleRegistry` class:
  - Constructor takes `{ role: 'api' | 'worker' | 'all', modules: string[] }`.
  - `private loaded = new Map<string, ModuleDefinition>()`
  - `private cqrs: CqrsBus`
  - `private eventBus: TypedEventBus`
  - `async loadAll(): Promise<void>` -- iterates config.modules, for each name:
    1. Dynamic import: `const mod = await import(\`@baseworks/module-${name}\`)`.
    2. Get definition: `const def: ModuleDefinition = mod.default ?? mod`.
    3. Validate that `def.name` matches the config name (log warning if mismatch).
    4. Register commands into cqrs bus: `for (const [key, handler] of Object.entries(def.commands ?? {})) { this.cqrs.registerCommand(key, handler) }`
    5. Register queries similarly.
    6. Store in loaded map.
    7. Log each loaded module.
  - `attachRoutes(app: Elysia<any>): void` -- if role is 'worker', return early. Otherwise iterate loaded modules and `.use(def.routes)` for each that has routes.
  - `getCqrs(): CqrsBus` -- returns the CQRS bus (for use in route handlers).
  - `getEventBus(): TypedEventBus` -- returns the event bus.
  - `getLoaded(): Map<string, ModuleDefinition>` -- for inspection/testing.

IMPORTANT on dynamic imports: If `import('@baseworks/module-example')` fails in Bun (see Research open question 1), implement a fallback static import map approach. Create a `moduleImportMap` constant:
```typescript
const moduleImportMap: Record<string, () => Promise<any>> = {
  example: () => import('@baseworks/module-example'),
  // Future modules added here
}
```
The loadAll method uses this map instead of string interpolation. This ensures the imports are statically analyzable by Bun.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Config -> Module Registry | Module names from config drive dynamic imports |
| Client -> Elysia Routes | HTTP request body is untrusted input |
| Command Input -> Handler | CQRS command input is untrusted until validated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Elevation of Privilege | apps/api/src/core/registry.ts | mitigate | Module names are validated against a static import map (moduleImportMap). Arbitrary strings cannot trigger arbitrary imports. Only pre-registered module names are loadable. |
| T-01-05 | Tampering | CQRS command input | mitigate | Every command handler uses `defineCommand` with TypeBox TypeCompiler validation. Invalid input returns VALIDATION_ERROR result, never reaches business logic. |
| T-01-06 | Tampering | HTTP request body | mitigate | Elysia route schemas validate request bodies via TypeBox before handler execution. Invalid bodies return 400. |
| T-01-07 | Denial of Service | apps/api/src/core/event-bus.ts | mitigate | Async subscriber errors are caught and logged, not propagated. A failing subscriber cannot crash the event bus or block other subscribers. |
</threat_model>
