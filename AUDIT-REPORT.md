### 🔴 [critical/correctness-bug] billing-webhook-behind-tenant-middleware
**File:** `apps/api/src/index.ts:300-317` · confidence: high · fix-risk: medium

**Problem:** The webhook endpoint's own docstring (routes.ts:23-37) states it must be public with 'no auth, no tenant middleware', but because the whole billing plugin is mounted AFTER the scoped tenantMiddleware derive, every request to POST /api/billing/webhooks first runs the tenant derive. Stripe/Pagar.me webhook POSTs carry no session cookie, so tenantMiddleware throws 'Unauthorized' before the webhook handler runs. Webhook ingestion is effectively broken: no subscription/payment events will ever be processed, silently stranding all billing state.

**Recommendation:** Split the public webhook route out of the tenant-scoped billing plugin and mount it in the public band (before tenantMiddleware).  1. In packages/modules/billing/src/routes.ts: extract the POST /webhooks handler into a new exported plugin `export const billingWebhookRoutes = new Elysia({ prefix: '/api/billing' }).post('/webhooks', ...)`. Keep only the tenant-scoped commands/queries (checkout, cancel, change, one-time, portal, usage, subscription, history) in `billingRoutes`.  2. Expose the webhook plugin from the module. Either add a field to the module definition in packages/modules/billing/src/index.ts (e.g. a `publicRoutes: billingWebhookRoutes`) and extend ModuleDefinition in @baseworks/shared + the registry to surface it, or export `billingWebhookRoutes` directly and import it in apps/api/src/index.ts.  3. In apps/api/src/index.ts, mount the webhook plugin in the public/operator band alongside authRoutes and /health — i.e. BEFORE `.use(tenantMiddleware)` (around line 298, same place authRoutes is mounted) — while leaving `.use(billingApiRoutes ?? new Elysia())` AFTER the scoped derive (line 317) for the tenant-scoped commands. Because the webhook route is fully self-contained (it reads raw body + provider signature, does its own DB idempotency + enqueue), it needs no handlerCtx/tenant context.  4. Add an integration test that POSTs to /api/billing/webhooks with NO session cookie and asserts the response is NOT 401/'UNAUTHORIZED' (it should reach signature verification and return 400 'Missing webhook signature header' for an unsigned request), proving the route is publicly reachable. Note: error.ts maps 'Unauthorized' to 401, not 500 as it appears in raw reproduction, so assert specifically against the UNAUTHORIZED code.

### 🔴 [critical/security] authz-owner-role-platform-admin-escalation
**File:** `apps/api/src/routes/admin.ts:38-39` · confidence: high · fix-risk: high

**Problem:** requireRole("owner") (packages/modules/auth/src/middleware.ts:79) checks the user's role *inside their own active organization*. Every user is auto-provisioned a personal organization on signup with creatorRole "owner" (auth.ts:86, databaseHooks user.create.after). Therefore EVERY authenticated user passes requireRole("owner") and gains access to the cross-tenant admin surface: GET /api/admin/tenants (list all organizations), GET /api/admin/users (all user emails), PATCH /api/admin/tenants/:id (edit any tenant), GET /api/admin/billing/overview (platform-wide MRR/subscriber data). The same flawed gate also fronts bull-board (apps/api/src/routes/bull-board.ts:77, all job payloads across tenants) and /health/detailed (health-detailed.ts:43, infra internals). There is no platform-admin / super-admin concept — per-tenant 'owner' is conflated with platform operator. This is broken access control / horizontal+vertical privilege escalation exposing all tenants' data to any registered user.

**Recommendation:** Introduce a platform-operator authorization distinct from the per-organization role. Concretely: (1) Add a boolean platform-admin marker — preferred as a new column on the better-auth user table (e.g. isPlatformAdmin boolean not null default false) via a Drizzle migration, or, if avoiding a migration for now, an env allow-list of operator user IDs/emails (e.g. PLATFORM_ADMIN_USER_IDS). (2) Add a new guard requirePlatformAdmin() in packages/modules/auth/src/middleware.ts modeled on requireRole but checking the platform-admin marker on session.user (NOT the membership role, and NOT requiring/consulting activeOrganizationId). It should throw Unauthorized (401) without a session and Forbidden (403) when the marker is absent. (3) Replace .use(requireRole("owner")) with .use(requirePlatformAdmin()) in apps/api/src/routes/admin.ts:39, apps/api/src/routes/bull-board.ts:77, and apps/api/src/routes/health-detailed.ts:43. Keep requireRole("owner") only for genuinely tenant-scoped operations (e.g. DELETE /api/tenant). (4) Provide a controlled way to designate the first platform admin (seed script or env allow-list) so admin tooling is not locked out after the change. (5) Add regression tests asserting a normal signed-up user (owner of their personal org but not a platform admin) receives 403 on /api/admin/*, /admin/bull-board, and /health/detailed, and that a designated platform admin receives 200. Update apps/api/src/__tests__/admin-auth.test.ts accordingly.

### 🟠 [high/correctness-bug] billing-occurredat-not-from-provider
**File:** `packages/modules/billing/src/adapters/stripe/stripe-webhook-mapper.ts:60` · confidence: high · fix-risk: medium

**Problem:** occurredAt is assigned the local wall-clock time at the moment of normalization (during ingestion), not the provider's event timestamp. This is exactly the uncoordinated, race-prone ordering the lastEventAt guard in handleSubscriptionUpdated was built to prevent.
